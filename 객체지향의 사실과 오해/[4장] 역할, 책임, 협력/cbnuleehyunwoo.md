# 4장 역할, 책임 협력

- 개인은 항상 자신에게 최대의 이익이 돌아올 수 있도록 합리적으로 의사 결정하는가?
    - 실상은 그렇지 않았다!
    - 인간의 본질의 특성에 따라 행동을 결정하는 것이 아니라, 어떤 상황에 처해있느냐가 인간의 행동을 결정했다.
- 결론: 문맥을 고려해야 한다, 객체 지향에서 문맥이란 바로 협력이다! 이를 고려하지 않고 객체가 가져야 할 상태와 행동부터 고민하면 안된다…
- 훌륭한 객체지향 설계란?
    - 절대 개별 객체의 품질이 훌륭한 객체지향 설계를 만들어내지 않는다.(오만한 객체라면 더더욱..!)
    - 조화를 이루며 적극적으로 상호작용 협력적인 객체를 창조하는 것,(각각의 객체가 조금 기묘하고 비합리적이더라도…)

## 협력

### 요청하고 응답하며 협력하는 사람들

- 협력의 본질은 무엇인가?
    - 요청과 응답으로 연결되는 사람들의 네트워크!
    - 협력은 다수의 연쇄적인 요청과 응답의 흐름으로 구성된다.

### 누가 파이를 훔쳤지?

- 협력이라는 단어 속에 내포된 다양한 특성을 두루 살펴보자(~~두루요?~~)
- <<이상한 나라의 앨리스>> 에서는 하트 잭에 대한 재판 장면이 등장한다…
    - 하트 잭을 재판하기 위해 재판에서는 다양한 인물들이 등장한다.
    - 즉 하트 잭의 재판이라는 동일한 목적을 위해 협력하고 있는 것이다!

### 재판 속의 협력

- 이를 객체지향의 관점에서 바라보자!
- 하트 잭의 재판 과정은 재판에 참여하는 사람드르이 요청과 응답에서 비롯된다.
    - 누군가가 왕에게 재판을 `요청`한다.
    - 재판 `요청`을 받은 왕이 하얀 토끼에게 증인을 부를 것을 `요청`한다.
    - 왕의 `요청`을 받은 토끼는 모자 장수에게 증인석으로 입장할 것을 `요청`한다.
    - 모자 장수는 증인석에 입장함으로써 토끼의 `요청`에 `응답`한다.
    - 모자 장수의 입장은 왕이 토끼에게 `요청`했던 증인 호출에 대한 `응답`일 것이다.
    - 모자 장수는 자신이 알고 있는 내용을 증언함으로써 왕의 `요청`에 `응답`한다.

## 책임

- 어떤 객체가 책임을 가진다?
    - 어떤 요청에 대답해 줄 수 있거나 적절한 행동을 할 의무가 있는 경우!
    - 앨리스의 이야기에서 왕은 재판을 수행하라 라는 요청에 응답해야 한다.
        - 왕은 재판을 수행할 책임이 있다…
    - 토끼의 경우에는 목격자를 불러오라는 요청에 응답해야한다.
        - 토끼는 목격자를 불러올 책임이 있다..
- 비자율적인 객체….
    - 변경에 취약하고 다양한 협력에 참여할 수 없는 객체이다…

### 책임의 분류

- 하는 것 vs 아는 것
    - 책임을 하는 것
        - 객체를 생성하거나 계산을 하는 등의 스스로 하는 것
        - 다른 객체의 행동을 시작시키는 것
        - 다른 객체의 활동을 제어하고 조절하는 것
    - 책임을 아는 것
        - 개인적인 정보에 관해 아는 것
        - 관련된 객체에 관해 아는 것
        - 자신이 유도하거나 계산할 수 있는 것에 관해 아는 것
- 왕의 재판 집행은 책임을 하는 것일까 아는 것일까?
    - 하는 것: 재판에 참여하는  다른 객체들의 활동을 제어하고 조율하므로…
- 하얀 토끼의 경우 책임을 하는 것일까 아는 것일까?
    - 아는 것: 목격자가 모자 장수라는 사실을 알고있다!
    - 하는 것: 모자 장수가 증인석에 입장하도록 요청한다!
    - 즉 둘다
- 모자 장수의 경우도 동일하다.
    - 아는 것: 증언에 대해 알고 있다.
    - 하는 것: 스스로 증인석에 입장해야 한다.
- 이렇듯 책임은 아는 것의 측면 즉 정보와 하는 것의 측면 즉 외부에 제공해 줄 수 있는 서비스의 목록이라고 할 수 있겠다.
- 책임은 객체의 공용 인터페이스를 구성한다.

### 책임과 메시지

- 메시지 전송이란 무엇인가?
    - 객체가 다른 객체에게 주어진 책임을 수행하도록 요청을 보내는 것
- 당연하게도 객체는 다른 객체로부터 요청이 전송됐을 경우에만 자신에게 주어진 책임을 수행한다.
- 책임은 협력이라는 문맥 속에서 요청을 수신하는 한 쪽의 객체 관점에서 무엇을 할 수 있는지에 대한 나열이다.
- 메시지는 관계이다
    - 왕은 `증언 하라` 라는 메시지를 전송할 수 있기 때문에, 메시지를 기반으로 왕과 모자 장수 사이의 상호 협력이 가능하다!
- 책임과 메시지의 수준은 같은가?
    - 같지 않다, 책임은 객체가 협력에 참여하기 위해 수행할 행위를 개략적으로 서술한 것이므로, 이를 협력을 통해 해결하기 위해 여러 메시지로 분할하는 것이 일반적이다!
- 객체지향 설계의 시작은….
    - 협력에 참여하기 위해 어떤 객체가 어떤 책임을 수행해야 하고 어떤 객체로부터 메시지를 수신할 것인지 결정하는 것부터…!

## 역할

### 책임의 집합이 의미하는 것

- 어떤 객체는 책임의 집합을 갖는다
    - ex> 모자 장수는 (증인석에 입장한다, 증언한다), 왕은 (재판한다, 목격자를 불러오도록 요구한다)
- 이런 것에 미루어서 봤을 때, 모자 장수는 증인, 왕은 판사 라는 역할을 수행하고 있음을 알 수 있다…
- 책임의 집합은 객체가 협력 안에서 수행하는 역할을 암시한다.
    - 이게 왜 중요한가?
        - 역할이 재사용 가능하고, 유연한 객체지향 설계를 도출하기 때문이다.
            
            > 흠… 그런가
            > 

### 판사와 증인

- 앨리스의 이야기는 다음과 같이 전개된다
    - 모자 장수가 증언을 마치고(증인의 역할을 수행하고) 다음 증인이 들어온다
    - 왕이 머리가 아파, 여왕이 판사의 역할을 수행한다.
- 위와 같이 역할을 수행하는 주체가 바뀐 상태에서 재판 과정이 바뀌는가?
    - 그렇지 않다.
        - 누군가 여왕에게 재판을 요청함으로써 재판이 시작된다.
        - 여왕이 하얀 토끼에게 증인을 부를 것을 요청한다.
        - 여왕의 요청을 받은 토끼는 앨리스에게 증인석으로 입장할 것을 요청한다.
        - …
- 협력 과정은 똑같지만, 주체도 다르므로.. 이 협력들을 구분하고 관리해야 할까?
    - 그렇지 않다!

### 역할이 답이다

- 그렇다면 이 중복을 어떻게 해소할 수 있을까?
    - 아주 간단하다. 판사와 증인이라는 역할을 사용하자
        - 누군가가 판사에게 재판을 요청함으로써 재판이 시작된다.
        - 판사는 하얀 토끼에게 증인을 부를 것을 요청한다.
        - 판사의 요청을 받은 토끼는 증인에게 증인석으로 입장할 것을 요청한다.
        - …
- 역할은 협력 내에서 다른 객체로 대체할 수 있음을 나타내는 일종의 표식이다!
    - 해당 역할을 수행할 수 있으면 어떤 객체라도 이를 대신할 수 있다…
- 앨리스가 판사의 역할을 대체할 수 있을까? 없다면 그 이유는 무엇인가.
    - 대체할 수 없다. 그 이유는 당연히 재판하라는 메시지를 동일하게 이해하고 처리할 수 없기 때문이다.
- 결론적으로…
    - 역할의 개념을 사용하면 유사한 협력을 추상화하여 인지 과부하를 줄일 수 있다.
    - 다양한 객체들이 동일한 협력에 참여할 수 있기에 재사용성이 높아진다.

### 협력의 추상화

- 협력의 추상화란 무엇인가?
    - 역할을 이용하여 하나의 협력 안에 여러 종류의 객체가 참여할 수 있게 하는 것이다…
- 협력의 추상화는 설계자가 다뤄야 하는 협력의 개수를 줄이는 동시에 구체적인 객체를 추상적인 역할로 대체한다.
    - ex>
        - 왕 - 토끼 - 모자 장수
        - 왕 - 토끼 - 요리사
        - 여왕 - 토끼 - 앨리스
    - 이는 사실 다음과 같이 추상화 할 수 있다.
        - 판사 - 토끼 - 증인
- 역할을 적절히 이용하여 협력을 단순화 하자!

### 대체 가능성

- 객체가 역할을 대체할 수 있는지를 그렇다면 어떻게 결정할 수 있겠는가?
    - 협력 안에서 역할이 수행할 수 있는 행동(책임)을 그대—로 수행할 수 있어야 한다.
- 객체는 그렇다면 수행하고 있는 역할에 배정된 행동만 수행할 수 있는가?
    - 아니다. 한 객체는 여러 역할을 지닐 수 있으므로..
        - ex> 왕은 재판, 국정.. 앨리스는 증인, 주인공…
- 그러므로 객체와 역할은 일반화/특수화 관계로 접근하는 것이 올바르다.
    - 역할은 일반화, 객체의 타입(역할)은 특수화!
- 결론적으로 역할이 대체 가능하다? 행위가 호환된다 즉 동일한 책임을 수행할 수 있다는 것!

## 객체의 모양을 결정하는 협력

### 흔한 오류

오류 1. 객체는 시스템에 필요한 데이터를 저장하기 위해 존재한다. 

- 객체의 목적을 시스템에 필요한 데이터를 저장하기 위해서…라고 착각하는 사람들이 많다.
    - 물론 객체가 데이터를 저장하기 때문에 100퍼센트 틀렸다고는 할 수 없지만, 데이터는 결국 객체가 행위, 즉 책임을 수행하기 위한 재료일 뿐이다…
- 결국 객체의 행동(책임)이 중요하다.

오류 2. 객체지향이 클래스와 클래스 간의 관계를 표현하는 시스템의 정적인 측면에 중점을 둔다. 

- 계속 강조하듯이 정적인 클래스가 아니라 동적인 객체에 중심을 두어야 한다. 클래스는 단순히 객체를 표현하고 생성하기 위한 프로그래밍 언어의 매커니즘이다…
- 예시
    - 객체를 독립적으로 바라보고 설계하기
        - 왕의 인스턴스를 다음과 같이 모델링하는 경우
            - 왕관을 쓰고…
            - 멋진 수염을 기르고…
            - 근엄한 표정으로…
        - 객체를 독립적으로 바라보면 위와 같은 불필요한 객체의 속성이 포함되기 쉽다.

### 협력을 따라 흐르는 객체의 책임

- 견고하고 깔끔한 협력
- 협력을 설계한다는 것
    - 설계에 참여하는 객체들이 주고받을 요청과 응답의 흐름을 결정한다는 것.
    - 요청의 응답과 흐름을 결정하다보면… 객체가 협력에 참여하기 위해 수행될 책임을 도출할 수 있다!
- 다음과 같은 프로세스일 것이다!
    - 협력 설계→ 요청과 응답의 흐름 결정 → 일련의 책임 고안 → 이 책임을 수행할 객체 선정
- 적어도 위 프로세스를 거치고 클래스를 개발해보자

## 객체지향 설계 기법

### 책임-주도 설계

- 협력에 필요한 책임들을 식별하고 적합한 객체에게 책임을 할당하는 방식으로 설계!
- 개별적인 객체의 상태가 아니라 객체의 책임과 상호작용에 집중하는 설계
- 시스템의 책임을 객체의 책임으로 변환, 각 객체가 책임을 수행하는 중에 필요한 정보나 서비스를 제공해줄 협력자를 찾는다…
    - 협력자를 찾았다면 해당 협력자에게 책임할당!
- 협조적이고 성실한 객체 시민들로 구성된 객체지향 시스템을 설계하는 절차
    - 시스템이 사용자에게 제공해야 하는 기능인 시스템 책임 파악
    - 시스템 책임을 더 작은 책임으로 분할
    - 이를 수행할 적절한 협력자(객체or역할)를 찾아 책임을 할당한다.
    - 객체가 책임을 수행하는 도중 다른 객체의 도움이 필요하면 이를 책임질 적절한 객체 또는 역할 찾기..
    - 결론적으로 여러 객체에 여러 역할, 책임이 할당되어 서로 협력하게 된다..!
- 결론적으로 역할, 책임, 협력은 유연하고 견고한 객체지향 시슽메을 만드는데 가장 중요한 재료, 그 외의 장치는 보조재료이다.

### 디자인 패턴

- 책임 주도 설계 vs 디자인 패턴
    - 책임 주도 설계는 객체의 역할, 책임, 협력을 고안하기 위한 방법과 절차를 제시하는 반면..
    - 디자인 패턴은 책임-주도 설계의 결과를 표현한다.
        
        > 원인과 결과 느낌이네요
        > 
- 패턴은 해결하고자 하는 문제가 무엇인지 명확하게 서술하고, 패턴을 적용할 수 있는 상황과 없는 상황을 함께 설명한다..
- 결론적으로 디자인 패턴은 공통으로 사용할 수 있는 역할, 책임, 협력의 템플릿이다.
    - 디자인 패턴을 잘 알고 있다면 (어떤 상황에 어떤 패턴을 적용할지)
    - 이전에 소개한 책임 주도 설계의 절차를 건너뛰고 손쉽게 역할-책임-협력 관계를 쉽게 표현할 수 있다.

### 테스트-주도 개발

- TDD가 응집도가 높고 결합도가 낮은 클래스로 구성된 시스템을 개발할 수 있게 하는 최상의 프랙티스는 맞다…
    - 하지만 객체지향에 대한 경험이 적다면 개발을 주도하기 위한 테스트 코드를 작성하는 것부터 어려움을 느끼므로.. 좋지 않다?
- TDD가 어렵다면 설계 경험과 패턴에 대한 지식이 없다는 것일 수도 있다. 즉 객체지향에 대한 깊이있는 지식을 습득하고… 시도해보면 좋을 것 같다.
