# 객체 지도

- 전통적인 소프트웨어 개발 방법은 변경이 빈번하게 발생하는 기능에 안정적인 구조를 종속시킨다
- 반면에 객체지향 개발 방법은 안정적인 구조에 변경이 빈번하게 발생하는 기능을 종속시킨다.

## 기능 설계 대 구조 설계

- 모든 소프트웨어 제품의 설계에는 두 가지 측면이 존재한다.
    - 기능 측면의 설계
        - 제품이 사용자를 위해 무엇을 할 수 있는가? 에 중점
    - 구조 측면의 설계
        - 제품의 형태가 어떠해야 하는지에 대해 초점
- 결국엔 두 가지를 함께 녹여내야 함,
- 소프트웨어가 사용자에게 가치있는 이유
    - 사용자에게 필요한 기능을 제공하기 때문
- 즉 훌륭한 기능은 훌륭한 소프트웨어를 만드는 충분조건
- 반면에 훌륭한 구조는 훌륭한 소프트웨어를 만드는 필요조건이다!
    - 사용자가 원하는 새로운 기능을 빠르게 추가할 수 있도록….
- 요구사항이 변경되지 않는다면 개발자의 삶은 좀 더 지루했을 것…
    - ~~아…아쉽네요~~
- 미래의 변경에 대비할 수는 있지만 미래의 변경을 예측할 수는 없다!
- 미래에 대비하는 가장 좋은 방법은 무엇일까?
    - 변경을 예측하는 것이 아니라 변경을 수용할 수 있는 선택의 여지를 설계에 마련해 놓는 것
    - 설계의 일차적인 목표는 변경에 소모되는 비용을 낮추는 것이다!
- 변경의 여지를 어떻게 남겨놓을 수 있겠는가?
    - 자주 변경되는 기능이 아닌 구조를 중심으로 설계하는 것!
- 왜 객체지향에는 변경의 여지가 많은가?
    - 안정적인 객체 구조를 바탕으로 시스템의 기능을 객체 간의 책임으로 분배하므로…
    - 객체의 구조에 집중하고 기능이 객체의 구조를 따르게 만든다!
- 중요: 사람들에게 길을 묻지 마라
    - ~~네이버 지도 키겠습니다..~~

## 두 가지 재료: 기능과 구조

- 기능은 사용자가 자신의 목표를 달성하기 위해 사용할 수 있는 시스템의 서비스이다.
- 구조는 시스템의 기능을 구현하기 위한 기반,
    - 기능 변경을 수용할 수 있을 정도로 안정적이어야 한다.
    - 변경에 강한 구조!
- 두 재료를 어떻게 구하는가?
    - 구조는 사용자나 이해 관계자들이 도메인에 관해 생각하는 개념과 개념들 간의 관계로 표현한다.
    - 기능은 사용자의 목표를 만족시키기 위해 책임을 수행하는 시스템의 행위로 표현한다.
- 유스케이스 모델링:
    - 일반적으로 기능을 수집하고 표현하기 위한 기법
    - 결과: 유스케이스
- 도메인 모델링:
    - 구조를 수집하고 표현하기 위한 기법
    - 결과: 도메인

## 안정적인 재료: 구조

### 도메인 모델

- 도메인: 사용자가 프로그램을 사용하는 대상 분야!
- 도메인 모델에서 모델이 한 대상을 단순화해서 표현한 것!
- 모델:
    - 지식을 선택적으로 단순화 하고 의식적으로 구조화한 형태
    - 소프트웨어가 목적하는 영역 내의 개념과 개념 간의 관계, 다양한 규칙이나 제약 등을 주의 깊게 추상화한 것이다.
    - 모델은 복잡성을 관리하는 기본적인 도구이다.
- 도메인 모델:
    - 소프트웨어가 목적하는 영역 내의 개념과 개념 간의 관계, 다양한 규칙이나 제약 등을 주의 깊게 추상화 한 것
    - 소프트웨어  개발과 관련된 이해관계자들이 도메인에 대해 생각하는 관점이다.
- 도메인 모델 쉽게 정의하기:
    - 도메인과 모델의 정의를 연결한다.
- 은행원에게 은행 도메인
    - 고객과 계좌 사이의 돈의 흐름
- 자동차 딜러의 자동차 도메인
    - 구매되는 자동차와 판매되는 자동차의 교환
- 소프트웨어의 도메인이 무엇이건 상관없이 항상 도메인과 관련된 사람들이 도메인을 바라보는 모델이 존재함!
- 도메인 모델은 단순히 다이어그램이라기 보다는 이해관계자들이 바라보는 멘탈모델임
- 멘탈 모델
    - 사람들이 자기 자신, 다른 사람, 환경, 자신이 상호작용하는 사물들에 대해 갖는 모형
- 사용자들은 자신의 멘탈 모델과 유사한 방식으로 제품이 반응하고 움직일 것이라고 기대한다
    - 훌륭한 디자인이란 사용자가 예상하는 방식에 따라 정확하게 반응하는 제품
- 사용자 모델과 디자인 모델이 동일하기에는 어려움이 있다.
    - 사용자와 설계자의 소통 어려움

### 도메인의 모습을 담을 수 있는 객체지향

- 소프트웨어에서도 마찬가지로 최종 코드는 사용자가 도메인을 바라보는 관점을 반영해야 한다.
    - 애플리케이션이 도메인 모델을 기반으로 설계되어야 한다.
- 도메인 모델은 결국
    - 사용자들이 도메인을 바라보는 관점이면서,
    - 설계자가 시스템의 구조를 바라보는 관점인 동시에
    - 소프트웨어 안에 구현된 코드의 모습 그 자체이다.
    - ~~뭐가이리많나요~~
- 따라서 도메인 모델의 세 가지 측면을 모두 모델링 할 수 있는 유사한 모델링 패러다임을 사용해야 한다.
    - 이게 객체지향!
- 이를 가능하게 하는 객체지향의 특성
    - 연결완전성
    - 표현적 차이

### 표현적 차이

- 소프트웨어 객체가 현실 객체를 왜곡한다고 하더라도..
    - 소프트웨어 객체는 현실 객체를 기반으로 구축된다!
- 표현적 차이(의미적 차이)
    - 소프트웨어 객체와 현실 객체 사잉의 의미적 거리
    - 핵심은 은유를 통해 이 두 차이를 최대한 줄이는 것
- 현실에 존재하지 않는 가상의 세계를 대상으로 하는 소프트웨어 도메인..
    - 결국 은유를 통해 투영해야 하는 대상은 사용자가 도메인에 대해 생각하는 개념을 기반으로!

### 불안정한 기능을 담는 안정적인 도메인 모델

- 도메인 모델을 기반으로 코드를 작성하는 두 번째 이유
    - 도메인 모델을 이 제공하는 구조가 상대적으로 안정적이기 때문..
- 왜 사용자의 도메인을 바라보는 관점을 소프트웨어 설계에 넣어야 하는가
    - 사용자가 도메인의 본질적인 측면을 가장 잘 이해하고 있기 때문이다.

## 불안정한 재료: 기능

### 유스케이스

- 기능적 요구사항
    - 시스템이 사용자에게 제공해야 하는 기능의 목록을 정리한 것
- 시스템이 사용자에게 기능을 제공하는 이유는 무엇인가?
    - 사용자들이 시스템을 통해 달성하고자 하는 목표가 존재하기 때문이다.
- 사용자는 자신의 목표를 달성하기 위해 시스템과 상호작용을 진행한다.
    - 사용자의 요청
    - 시스템의 응답
    - 사용자의 요청…
    - 사용자와 시스템 사이의 상호작용은 사용자의 목표를 만족시키거나 에러 등의 이유로 상호작용을 더 이상 진행 할 수 없을 때 까지!
- 유스케이스:
    - 사용자의 목표를 달성하기 위해 사용자와 시스템 간에 이뤄지는 상호작용의 흐름을 텍스트로 정리한 것
- 유스케이스의 가치
    - 사용자들이 목표를 중심으로 시스템의 기능적인 요구사항들을 이야기 형식으로 묶을 수 있다는 점

### 유스케이스의 특성

1. 유스케이스는 사용자와 시스템 간의 상호작용을 보여주는 텍스트이다.
    - 유스케이스는 다이어그램이 아니다.
    - 중요한 것은 유스케이스 안에 포함되어 있는 상호작용의 흐름
2. 유스케이스는 하나의 시나리오가 아니라 여러 시나리오들의 집합이다. 
    - 시나리오:
        - 유스케이스를 통해 시스템을 사용하는 하나의 특정한 이야기 혹은 경로
        - 유스케이스 인스턴스라고도 함
3. 유스케이스는 단순한 피처(feature) 목록과는 다르다
    - 피처:
        - 시스템이 수행해야 하는 기능의 목록들을 단순하게 나열한 것
    - 유스케이스는 단순한 기능 나열이 아니라 이야기를 통해 연관된 기능들을 함께 묶을 수 있다는 것
4. 유스케이스는 사용자 인터페이스와 관련된 세부 정보를 포함하지 않아야 한다. 
    - 유스케이스는 자주 변경되는 사용자 인터페이스 요소는 배제하고 사용자 관점에서 시스템의 행위에 초점을 맞춤
5. 유스케이스는 내부 설계와 관련된 정보를 포함하지 않는다. 
    - 유스케이스의 목적:
        - 연관된 시스템의 기능을 이야기 형식으로 모으는 것, 내부 설계를 설명하는 것이 아님!

### 유스케이스는 설계 기법도, 객체지향 기법도 아니다

- 유스케이스는 단지 사용자가 바라보는 시스템의 외부 관점만을 표현한다.
- 유스케이스로부터 시스템의 내부 구조를 유추할 수 있는 방법은 존재하지 않는다.
- 심지어 유스케이스는 객체지향과도 상관이 없다.
    - 유스케이스와 객체 사이에는 커다란 간격이 존재한다.
        - 심지어 유스케이스와 객체의 간격을 자동으로 없앨 수 있는 방법도 존재하지 않고
        - 유스케이스를 기반으로 객체의 구조를 쉽게 추출할 수 있다는 것도 어설픈 설명이다.
        - 유스케이스는 객체의 구조나 책임에 대한 어떤 정보도 제공하지 않음
        - ~~오늘 제 유스케이스가 무너졌습니다.…~~

## 재료 합치기: 기능과 구조의 통합

### 도메인 모델, 유스케이스, 그리고 책임-주도 설계

- 불안정한 기능을 안정적인 구조 안에 담음으로써 변경에 대한 파급효과를 최소화 하는 것 → 훌륭한 객체지향 설계자가 갖춰야 할 기본적인 소양!
- 안정적인 구조를 개념화 하기 위해: 도메인 모델
- 불안정한 기능을 서술하기 위해: 유스케이스
- 객체지향 패러다임은 모든 것이 객체라는 사실에서 출발
    - 때문에 유스케이스에 명시된 기능을 구현하는 프로그래머는…
        - 시스템을 사용자로부터 전송된 메시지를 수행하기 위해 책임을 수행하는 하나의 거대한 자율적인 객체로 본다.
        - 시스템이 수행하는 커다란 규모의 책임은 시스템 안에 들어가는 더 작은 크기의 객체들의 협력을 통해 구현될 수 있다.

### 기능 변경을 흡수하는 안정적인 구조

- 도메인 모델을 기반으로 객체 구조를 설계하는 이유가 무엇인가?
    - 도메인 모델이 안정적이기 때문!
- 도메인 모델이 왜 안정적인가?
    - 도메인 모델을 구성하는 요소가 다음과 같은 특징을 띠기 때문
        - 도메인 모델을 구성하는 개념은 비즈니스가 없어지거나 완전히 개편되지 않는 한 안정적으로 유지된다! ex> 정기예금, 계좌, 이자율
        - 도메인 모델을 구성하는 개념 간의 관계는 비즈니스 규칙을 기반으로 하기 떄문에  비즈니스 규칙이 변경되지 않는 한 안정적으로 유지..
- 객체지향의 연결완전성:
    - 도메인 을 모델링 하기 위한 기법과
    - 도메인을 프로그래밍 하기 위해 사용되느 기법이 닽다.
    - 즉 도메인 모델링에 사용된 객체와 개념을 프로그래밍 설계에서의 객체와 클래스로 매끄럽게 바꿀 수 있다!
- 이의 역도 성립한다, 가역성
    - 코드의 변경으로부터 도메인 변경 사항을 유추할 수 있다는 것…
