# 5장 책임과 메시징

## 자율적인 책임

### 설계의 품질을 좌우하는 책임

- 객체지향 공동체를 구성하는 기본적인 단위는 자율적인 객체이다.
- 중요한 것은 자율성
    - 자율성이 정확히 무엇인가?
        - 자기 스스로의 원칙에 따라 어떤 일을 하거나 자신을 통제해서 절제하는 성질
    - 때문에 타인이 정한 규칙이나 명령에 따라 판단하고 행동하는 객체는 자율적인 객체라고 부르기 어렵다…
- 객체가 어떤 행동을 왜 하는가?
    - 다른 객체로부터 요청을 수신했기 때문!

### 자신의 의지에 따라 증언할 수 있는 자유

- 객체가 책임을 자율적으로 수행하기 위해서는 초당부터 할당되는 책임이 자율적이어야 한다….
- 앨리스 이야기의 예시
    - 모자 장수는 왕의 요청을 받아 증언이라는 책임을 수행하나, 증언 방식이나 증언에 필요한 자료는 스스로의 판단에 따라 선택할 수 있다.
    - 하지만, 왕이 모자 장수에게
        - 목격했던 장면을 떠올려라
        - 떠오르는 기억을 시간 순서대로 기술하라
        - 말로 간결하게 표현하라

### 너무 추상적인 책임

- 하지만 모자장수가 왕에게 단순히, 설명하라 라고 메시지를 받았다면 모자 장수는무엇을 말해야 할지 모른다..
- 떄문에 책임은 협력에 참여하는 의도를 명확하게 설명할 수 있는 수준 안에서 추상적이어야 한다.

### 어떻게가 아니라 무엇을

- 자율적인 책임의 특징
    - 어떻게 해야하는가가 아니라 무엇을 해야하는가를 설명하는 것
- 이전에 언급한 모자장수와 왕의 예시에서 어떻게 ~~ 하라 라는 책임은 좋지 않았던 것처럼..

### 책임을 자극하는 메시지

- 책임이라는 말 속에는 어떤 행동을 수행한다. 라는 의미 내포
- 결국 어떤 객체에게 책임, 행동을 수행하게 만드는 것은 메시지이다.

## 메시지와 메서드

### 메시지

- 하나의 객체는 다른 객체에게 어떻게 접근하는가?
    - 메시지를 전송함으로써…
- 메시지 이름: 왕이 모자 장수에게 전송하는 메시지를 가리키는 `증언하라`
- 메시지를 전송할 떄 추가적인 정보가 필요한 경우 메시지의 인자를 통해 추가 정보를 제공할 수 있다!
    - ex> 특정한 장소의 특정한 시간 등등
- 결국 메시지는 메시지 이름과 인자, 두 부분으로 구성된다!
    - `증언하라(어제, 왕국)`
- 메시지 전송은 수신자와 메시지의 조합
    - `모자장수, 증언하라(어제, 왕국)`
- 메시지를 수신받은 객체는 우선 자신이 해당 메시지를 처리할 수 있는지 확인!
- 모자장수가 메시지를 변경하지만 않는다면, 책임을 수행하는 방법을 변경하더라도 왕은 알 길이 없음,
    - 즉 객체의 내부와 외부가 메시지를 기준으로 분리된다..

### 메서드

- 메서드: 모자 장수가 메시지를 처리하기 위해 내부적으로 선택하는 방법
- 객체는 메시지를 수신하면 먼저 해당 메시지를 처리할 수 있는지 여부를 판단
- 처리 가능하다면 주어진 책임을 다하기 위해 메세지를 처리할 메서드를 선택한다.
- 메시지는 어떻게 수행될 것인지 명시하지 않는다, 단지 무엇이 실행되기를 바라는지만..
- 메시지를 수신한 객체가 실행 시간에 메서드를 선택할 수 있다는 것음 사실
    - 다른 프로그래밍 언어와 객체지향 언어를 구분짓는 핵심적인 특징 중 하나임…

### 다형성

- 다형성
    - 서로 다른 유형의 객체가 동일한 메시지에 대해 다르게 반응하는 것
    - 즉 서로 다른 타입에 속하는 객체들이 동일한 메시지를 수신하면 서로 다른 메서드로 처리한다
- 앨리스 이야기에서 사실 모자장수, 토끼, 앨리스 등등은 왕의 증언하라 라는 메시지를 받은 것이다.
    - 이들은 자신만의 방법으로 증언에 도움을 준다.
- 다형성은 역할,책임,협력과 깊은 관련이 있음….
    - 동일한 역할을 수행할 수 있는 객체들 사이의 대체 가능성..이라던가

### 유연하고 확장 가능하고 재사용성이 높은 협력의 의미

- 왕은 증언하라는 메시지를 수신할 수 있지만, 구체적인 타입 즉 어느 수신자가 메시지를 수신할지 상관하지 않는다.
    - 즉 왕은 오직 수신자가 메시지를 이해할 수 있다는 사실만 알고 있는 상태에서 협력에 참여
- 왜 이렇게 했는가?
    1. 협력이 유연해짐.
        - 송신자는 수신자가 메시지를 이해만 한다면 누구라도 상관하지 않기 때문에 수신자를 다른 타입의 객체로 대체하더라도 송신자는 알지 못함
    2. 협력이 수행되는 방식을 확장할 수 있다. 
        - 즉 협력의 세부적인 수행 방식을 쉽게 수정할 수 있는데,
        - 이는 곧 재판장에 출석하지 않고도 동영상만 보내는 객체더라도 수용될 수 있다는 것..
    3. 협력이 수행되는 방식을 재사용할 수 있다. 
        - 다양한 문맥에서 해당 협력을 재사용해볼 수 있다.

### 송신자와 수신자를 약하게 연결하는 메시지

- 메시지는 송신자와 수신자 사이의 결합도를 낮춤으로써 설계를 유연하고, 확장 가능하고, 재사용 가능하게 만든다.
    - 송신자는 메시지만 바라본다, 단지 수신자가 메시지를 이해하고 잘 처리해줄 것이라는 사실만 알아도 충분…

## 메시지를 따라라

### 객체지향의 핵심, 메시지

- 클래스가 코드를 구현하기 위해 사용할 수 있는 중요한 추상화 도구인 것은 사실이나,
- 객체지향의 강력함은 객체들이 주고받는 메시지로부터 나옴
- 클래스를 중심에 두는 설계는 유연하지 못하고 확장하기 어려움
    - 진정한 객체지향 패러다임으로의 도약은 개별적인 객체가 아니라 메시지를 주고 받는 객체들 사이의 커뮤니케이션에 초점을 맞출때….

### 책임-주도 설계 다시 살펴보기

- 어떤 객체가 책임을 완수하기 위해 자신이 보유하고 있지 않은 정보를 필요로 하게 된다면 어떻게 해야 하는가?
    - 해당 정보를 제공할 책임을 담당하고 있는 객체에게 메시지를 전송해 정보를 요청해야 한다.
- 결과적으로는 메시지가 수신자의 책임를 결정한다!

### What/Who 사이클

- 책임 - 주도 설계의 핵심은 어떤 행위가 필요한지를 먼저 결정한 후에 이 행위를 수행할 객체를 결정하는 것
- 객체 사이의 협력 관계를 설계하기 위해 다음과 같은 프로세스
    - 어떤 행위를 수행할 것인지를 결정한 후에
    - 누가 그 행위를 수행할 것인지를 결정
- 객체의 행위를 결정하는 것은 객체 자체의 속성이 아니다!

### 묻지 말고 시켜라

- 다른 말로 디미터의 법칙
- 메시지를 먼저 결정하고 객체가 메시지를 따르게 하는 설계 방식
- 책임 주도 설계는 객체가 아니라 메시지에 초점을 맞추므로,, 객체지향의 장점을 극대화
- 메시지를 결정하는 시점에서 어떤 객체가 메시지를 수신할 것인지를 알 수 있는가?
    - No
    - 메시지 송신자는 메시지를 수신할 객체의 내부 상태를 볼 수 없음..
    - 따라서 메시지 중심의 설계는 메시지 수신자의 캡슐화를 증진시킺다.
- 묻지말고 시켜라 스타일은 객체지향 앱의 자율적인 객체들의 공동체라는 사실을 강조
- 결론적으로 묻지말고 시켜라 스타일은 객체를 자율적으로 만들고 캡슐화를 보장하여 결합도를 낮게 유지시킨다.

### 메시지를 믿어라

- 메시지를 이해할 수만 있다면, 다양한 타입의 객체로 협력 대상을 자유롭게 교체할 수 있음!
- 메시지를 믿으면 자율적인 책임은 저절로 따라온다
    - 메시지를 믿으면 광명을 찾을 수 있다

## 객체 인터페이스

### 인터페이스

- 인터페이스란 어떤 두 사물이 마주치는 경계 지점에서 서로 상호작용할 수 있게 이어주는 방법이나 장치
- 일반적으로 인터페이스는 다음과 같은 세 가지 특징을 갖는다
    1. 인터페이스의 사용법을 익히기만 하면, 내부 구조나 동작 방식을 몰라도 조작과 의사전달이 가능하다.
    2. 인터페이스 자체는 변경하지 않고 내부 구성이나 작동 방식만을 변경하는 것은 사용자에게 어떤 영향도 없다.
    3. 대상이 변경되더라도 동일한 인터페이스를 제공하기만 하면 문제없이 상호작용이 가능하다.

### 메시지가 인터페이스를 결정한다

- 객체 간의 유일한 상호작용은 메시지 전송이다.
- 떄문에 객체의 인터페이스는 객체가 수신할 수 있는 메시지의 목록으로 구성된다.

### 공용 인터페이스

- 모든 인터페이스의 모든 부분은 외부에서 접근할 수 있는가?
    - NO
- 사실 인터페이스는 공개된 인터페이스와 감춰진 인터페이스로 구분된다.
- 모자 장수의 공용 인터페이스는 무엇인가?
    - 왕이 모자 장수와 협력할 수 있는 수단은 증언하라 뿐이므로
    - 증언하라 라는 메시지를 전송하고 수신할 수 있는 인터페이스가 존재하는 것..
    - 다르게 말하면 모자 장수는 왕에게 증언하라는 메시지와 증인석에 입장하라 라는 메시지에 응답할 수 있는 인터페이스를 공개하고 있다…
- 객체는 인터페이스를 통해 메시지가 수신되었을 때, 증언하거나 증인석에 입장할 책임을 자율적으로 수행한다.

### 책임, 메시지, 그리고 인터페이스

- 지금까지의 여정 되짚기
1. 협력에 참여하는 객체의 책임이 자율적이어야 한다는 사실을 강조 
    - 자율성이란 자신의 의지와 판단력을 기반으로 객체 스스로 책임을 수행하는 방법을 결정할 수 있음을 의미,
2. 한 객체가 다른 객체에게 요청을 전송할 때 사용하는 매커니즘인 메시지
    - 객체의 인터페이스는 객체가 수신할 수 있는 메시지의 목록으로 채워진다.
3. 객체가 책임을 수행하기 위해 외부로부터 메시지를 받기 위한 통로인 인터페이스의 개념

## 인터페이스와 구현의 분리

### 객체 관점에서 생각 하는 방법

- 좀 더 추상적인 인터페이스
    - 상세한 수준의 메시지를 보내는 것은 객체의 자율성을 저해한다.
    - 목격했던 장면을 떠올려라, 재구성하라, 말하라… 등등보다는 증언하라
- 최소 인터페이스
    - 외부에서 사용할 필요가 없는 인터페이스는 최대한 노출하지 말라
- 인터페이스와 구현 간에 차이가 있다는 점을 인식

### 구현

- 객체지향의 세계에서 내부 구조와 작동 방식을 가리키는 고유의 용어
- 객체를 구성하지만 공용 인터페이스에 포함되지 않는 모든 것이 구현에 포함

### 인터페이스와 구현의 분리 원칙

- 훌륭한 객체란 무엇인가?
    - 구현을 모른 채 인터페이스만 알면 상호작용할 수 있는 객체를 의미
- 즉 객체를 설계할 때 객체 외부에 노출되는 인터페이스와 객체의 내부에 숨겨지는 구현을 명확하게 분리해서 고려해야한다.
- 인터페이스와 구현의 분리 원칙이 왜 중요한가?
    - 소프트웨어는 항상 변경되기 때문에…

### 캡슐화

- 객체의 자율성을 보존하기 위해 구현을 외부로부터 감추는 것.
- 캡슐화의 두가지 관점
    - 상태와 행위의 캡슐화
        - 객체는 상태와 행위의 조합이다.
        - 객체는 스스로 자신의 상태를 관리하며 상태를 변경하고 외부에 응답할 수 있는 행동을 내부에 함께 보관
    - 사적인 비밀의 캡슐화
        - 객체 역시 개인적인 비밀이 노출되는 것에 민감
        - 캡슐화를 통해 변경이 빈번하게 일어나는 불안정한 비밀을 안정적인 인터페이스 뒤로 숨길 수 있음,

## 책임의 자율성이 협력의 품질을 결정한다

- 객체들이 동일한 목적을 달성하기 위해 협력하는 방법의 가짓수는 수도 없이 많을 수 있음…
- 10명의 사람을 모아놓고 재판을 위한 협력 과정을 그려보라고 하면 각각 다른 협력과정을 내놓을 것….
- 결과적으로 책임이 얼마나 자율적인지가 전체적인 협력의 설계 품질을 결정하게 된다.
    1. 자율적인 책임은 협력을 단순하게 만든다.
    2. 자율적인 책임은 모자장수의 외부와 내부를 명확하게 분리한다. 
    3. 책임이 자율적인 경우 책임을 수행하는 내부적인 방법을 변경하더라도 외부에 영향을 미치지 않는다.
    4. 자율적인 책임은 협력의 대상을 다양하게 선택할 수 있는 유연성을 제공한다. 
    5. 객체가 수행하는 책임들이 자율적일 수록 객체의 역할을 이해하기 쉬워진다.
