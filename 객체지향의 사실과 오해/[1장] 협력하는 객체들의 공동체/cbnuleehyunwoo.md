# 1. 협력하는 객체들의 공동체

## 협력하는 사람들

#### 요청과 협력으로 구성된 협력

- 일상에서 발생하는 어떤 문제는 대부분 개인 혼자 해결하기에 복잡한 경우가 많다.
- 사람들은 이 때문에 다른 사람의 도움을 받아 문제를 해결하는 것을 선호한다.

#### 요청은 연쇄적으로 발생한다.

- 일반적으로 하나의 문제를 해결하기 위해, 다수의 사람 혹은 역할이 필요하므로, 한 사람의 요청이 다른 사람으로, 그리고 그 요청을 받은 사람이 다른 사람에게 요청하는.. 형태가 발생한다.

#### 응답은 요청의 방향의 반대로 연쇄적으로 전달된다.

- 요청을 받은 사람은 주어진 책임을 다한다, 필요한 지식이나 서비스를 전달하는 식으로..

#### 특정한 역할은 특정한 책임을 암시한다.

- 선생님은 학생을 가르칠 책임을, 경찰관은 범죄자를 검거할 책임이 있다..

#### 사람들이 특정한 역할을 맡고 책임을 수행한다는 사실은 다음과 연관된다.

- 여러 사람이 동일한 역할을 수행할 수 있다.
    - 손님 입장에서는 자신이 주문한 커피를 마시기만 하면 된다! 누가 주던 상관하지도, 궁금해 하지도 않을 것이다.
    - 캐시어도 마찬가지
- 역할은 대체 가능성을 의미한다.
    - 캐시어가 여러 명 있다면, 손님은 누가 캐시어 역할을 해주던 상관하지 않는다.
- 책임을 수행하는 방법은 자율적으로 수행할 수 있다.
    - 바리스타는 결과물만 괜찮다면, 커피를 자신이 원하는 대로 만들어도 된다!
- 한 사람이 동시에 여러 역할을 수행할 수 있다.
    - 캐시어가 바리스타의 역할을 수행할 수도 있을 것!

## 역할, 책임, 협력

#### 기능을 구현하기 위해 협력하는 객체들

- 위에서 정리한 내용은 객체지향의 핵심적이고 중요한 개념을 대부분 포함한다!

#### 역할과 책임을 수행하며 협력하는 객체들

- 객체의 역할은 사람의 역할과 유사하게 다음과 같은 특징을 지닌다
    - 여러 객체가 동일한 역할을 수행할 수 있다.
    - 역할은 대체 가능성을 의미한다.
    - 각 객체는 책임을 수행하는 방법을 자율적으로 선택할 수 있다.
    - 하나의 객체가 동시에 여러가지 역할을 수행할 수 있다.
- 왜 사람을 객체로 바꾸면 위의 설명이 객체지향이 되는가?
    - 인간은 사회적 동물이다. 즉 사람도 특정한 목표를 이루기 위해 서로 협력하는 경우가 많으므로,
    - 하나의 목표는 작은 책임으로 분할되어 적절한 역할을 가진 사람에 의해 수행된다!
        - 이러한 과정에서 서로 도움을 요청하기도 하고 이에 대한 응답도 오가는 협력 관계가 완성된다!
    - 그렇다면 반대로 객체는 왜 인간의 세계와 유사한가?
        - 워드 커닝험의 `어떤 객체도 섬이 아니다`
        - 사람이 최종적으로 인식하게 되는 시스템(목표)도 사실 객체들이 성실하게 협력해서 일궈낸 결실

## 협력 속에 사는 객체

- 역할, 책임, 협력이 객체지향 애플리케이션의 윤곽을 결정한다.
- 그렇지만 이러한 역할, 책임, 협력을 실제로 수행하는 것은 객체이다.
- 인간 세계에서 사람이 없으면 역할, 책임, 협력이 아무 의미가 없는 것처럼, 객체지향 세계 역시 객체가 중심이다.
- 객체의 품질이 협력의 품질을 결정한다.,
    - 객체는 애플리케이션의 기능을 구혀하기 위해 존재하지만, 작은 기능 하나도, 객체 하나가 감당하지 못하는 경우가 더 많다…
- 협력 공동체의 일원으로서 객체는 다음과 같은 두 가지 덕목을 갖춰야 한다
    1. 객체는 충분히 협력적이어야 한다. 
        - 객체는 다른 객체의 요청에 귀 기울이며, 다른 객체에게 적극적으로 도움을 요청할 정도로 열린 마음을 지녀야 한다.
        - God object
            - 외부의 도움을 무시한 채 모든 것을 스스로 처리하려고 하는 전지전능한 객체
            - 내부적인 복잡도에 의해 자멸한다
            
            > God object가 어떤 문제가 있으리라는 것은 알겠는데, 어떤 문제가 구체적으로 있을지 잘 안와닿네요
            > 
        - 협력적이다 라는 것은 다른 객체의 명령에 따라 행동하는 수동적인 존재가 아님, 즉 시킨 거 하는 게 아니라는 것!
            - 그냥 요청에 응답하는 것으로 생각하면 될 듯, 즉 어떻게 응답할지는 요청받은 객체가 알아서..
            - 심지어 요청에 응하지 않을 수도 있음(MZ 객체네요?)
    2. 객체는 충분히 자율적이어야 한다. 
        - 캐시어가 바리스타에게 커피를 만들어 달라고 할 때, 바리스타가 어떤 방식으로 커피를 만들라는 구체적인 프로세스를 제시하지는 않는다.

#### 상태와 행동을 함께 지닌 자율적인 객체

- 객체가 협력에 참여하기 위해 어떤 행동을 해야한다 → 그 행동을 하는 데 필요한 상태를 가지고 있어야 한다,
    - 즉 객체가 어떤 행동을 하기 위해 필요한 상태를 객체가 알고 있어야 한다, 그래야 객체 스스로 판단하고 결정할 수 있지 않겠는가.
- 객체의 자율성:
    - 객체의 자율성은 객체의 내부와 외부를 명확하게 구분하는 것에서 나온다.
    - 객체의 사적인 부분은 객체 스스로 관리하고, 외부에서 일체 간섭할 수 없게 차단하여야 한다.
- 객체는 다른 객체가 무엇을 수행하는지 알 수는 있다.
- 단 다른 객체가 무엇을 어떻게 수행하는지 까지는 알 수 없다, 알 필요도 없다.

#### 협력과 메시지

- 인간 세계에서는 협력을 위해 다양한 수단을 사용했다!(말, 글, 제스쳐 등등…)
- 그렇지만 객체 지향 세계에서는 단 한 가지의 의사소통 방법만이 존재한다.
    - 메시지:
        - 메시지 전송: 한 객체가 다른 객체에게 요청하는 것
        - 메시지 수신: 한 객체가 다른 객체에게 요청을 받는 것
    - 송신자, 수신자
        - 송신자: 메시지를 전송하는 객체
        - 수신자: 메시지를 수신하는 객체

#### 메서드와 자율성

- 그렇다면 메시지를 어떻게 이해할 수 있겠는가? 우선 수신자는 먼저 수신된 메시지를 이해할 수 있는지의 여부를 먼저 판단한다.
- 미리 정해진 자신만의 방법에 따라 메시지를 처리한다!
    - 이를 메서드라 한다.
- 메시지를 수신한 객체는 실행시간에 메서드를 선택할 수 있다
    - → 프로시저 호출에 대한 실행 코드를 컴파일 시간에 결정하는 절차적인 언어와 확연히 구분된다.
        
        > 근데 절차적인 언어라고 꼭 프로시저 호출이 컴파일 시간에 결정되나요??
        > 
- 메시지와 메서드의 분리는 객체들 간의 자율성을 증진시킨다.

## 객체지향의 본질

- 그래서 객체지향이 무엇인가?
    - 객체지향이란 시스템을 상호작용하는 자율적인 객체들의 공동체로 바라보고 객체를 이용해 시스템을 분할하는 방법이다.
    - 자율적인 객체란 상태와 행위를 함께 지니며, 스스로 자기 자신을 책임지는 객체를 의미한다.
    - 객체는 시스템의 행위를 구현하기 위해 다른 객체와 협력한다. 각 객체는 협력 내에서 정해진 역할을 수행하며 역할은 관련된 책임의 집합이다.
    - 객체는 다른 객체와 협력하기 위해 메시지를 전송하고, 메시지를 수신한 객체는 메시지를 처리하는 데 적합한 메서드를 자율적으로 선택한다.

#### 객체를 지향하라

- 에스키모인들은 눈에 대한 어휘를 400여개정도 가지고 있다는 소문이 퍼진적이 있다. 물론 이는 사실이 아니다.
- 이와 비슷하게, 객체지향과 클래스도 과장되어 전달되는 감이 있다.
    - 객체지향의 인기를 주도한 SmallTalk나, C++, 자바의 초점은 새로운 개념의 데이터 추상화를 제공하는 클래스라는 빌딩 블록에 맞춰져 있었지만,
    - 다양한 언어가 출현하고 다양한 서적이 출간되면서 점점점점 클래스 자체에 이목이 집중된 감이 없지 않아 있다
    - 많은 사람들이 객체지향이라는 말을 들으면 조건반사적으로 클래스를 떠올린다.
- 물론 클래스가 객체지향 프로그래밍 언어의 관점에서 매우 중요한 요소이지만, 이정도로 강조되는 것에는 약간의 문제가 있다.
- 결론적으로 저자가 전하고자 하는 포인트는 다음과 같다.
    - 지나치게 클래스를 강조하는 프로그래밍 언어적인 관점은 객체의 캡슐화를 저해하고 클래스를 강하게 결합시킨다.
    - 훌륭한 객체지향 설계자가 되려면, 코드를 담는 클래스의 관점에서 메시지를 주고받는 객체의 관점으로 사고의 중심을 전환해야 한다.
    - 클래스는 객체들의 협력 관계를 코드로 옮기는 도구에 불과하다!
        - like 객체 지향세계의 도시전설
    - 그러므로 클래스의 구조와 메서드에 집중하는 것보다 객체의 역할, 책임. 협력에 집중하는 것이 더 좋다!
