# 함께 모으기

## 커피 전문점 도메인

### 커피 주문

### 커피 전문점이라는 세상

- 커피 전문점 안에는 메뉴판이 존재한다.
    - 아메리카노, 카푸치노, 카라멜 마끼아또, 에스프레소
- 객체지향의 관점에서
    - 메뉴판은 하나의 객체이다
    - 메뉴 항목 역시 객체이다
    - 메뉴판은 그렇다면 4개의 항목 객체를 포함하는 객체이다
- 손님은 메뉴판을 보고
- 바리스타에게 원하는 커피를 주문한다.
- 객체지향의 관점에서
    - 손님도, 바리스타도 하나의 객체이다
    - 때문에 손님 객체는 메뉴판 객체 안에 적힌 메뉴 항목 객체들 중에서 자신이 원하는 메뉴 항목 객체 하나를 선택하여 바리스타 객체에게 전달한다.
    - 바리스타는 주문을 받은 메뉴에 대해서 적절한 커피를 제조한다.
    - 바리스타는 자율적으로 커피를 제조한다.
    - 제조되는 커피 또한 객체이다.
- 커피숍 세상의 객체 리스트는 다음과 같다.
    - 손님 객체
    - 메뉴 항목 객체
    - 메뉴판 객체
    - 바리스타 객체
    - 커피 객체
- 객체들 간의 관계 살펴보기
    - 손님은 메뉴판에서 주문할 커피를 선택할 수 있어야 한다.
        - 따라서 손님은 메뉴판을 알아야 한다. (관계가 존재한다)
    - 손님은 바리스타에게 주문을 할 수 있어야 한다.
        - 따라서 손님은 바리스타를 알아야 한다. (관계가 존재한다)
    - 바리스타는 커피를 제조해야 한다.
        - 따라서 바리스타는 커피를 알아야 한다.(관계가 존재한다)
- 손님 객체는 손님 타입의 인스턴스이고, 바리스타 객체는 바리스타 타입의 인스턴스라고 볼 수 있다.
- 아메리카노, 에스프레소, 카라멜 마끼아또, 카푸치노 등등 또한 커피 타입의 인스턴스이다.
- 합성(포함) 관계 vs 연관 관계
    - 메뉴 항목은 메뉴판에 포함되므로 메뉴 항목 타입과 메뉴판 타입은 합성 관계이다.
    - 하지만 손님 타입과 메뉴판 타입은  연관 관계이다, 메뉴판이 손님의 일부가 아니므로..
    - 바리스타는 커피를 제조해야 하므로, 바리스타 타입과 커피 타입도 연관관계이댜.
- 위에 이렇게 쭉 나열해 보았지만 실제로 도메인 모델을 설계할 때 어떤 관계가 포함인지, 연관인지는 크게 중요하지 않다고 한다!
    - 초점은
        - 어떤 타임이 도메인을 구성하는가?
        - 타입들 사이에 어떤 관계가 존재하는가?
        - 도메인을 이해하는 것

## 설계하고 구현하기

### 커피를 구현하기 위한 협력 찾기

- 객체지향 설계의 첫 번째 목표!
    - 훌륭한 객체를 설계하는 것 보다 훌륭한 협력을 설계하는 것
- 협력을 설계할 때는 객체가 메시지를 선택하는 것이 아니라 메시지가 객체를 선택하게 해야 함!
    - 메시지를 먼저 선택하고 그 후에 메시지를 수신하기에 적절한 객체를 선택
- 첫 번째 메시지 커피를 주문하라
- 우선 메시지를 처리하기에 적합한 객체를 찾자
    - 도메인 모델 안에 책임을 수행하기에 적절한 타입이 존재하는지 찾아보자
    - 적절한 타입을 찾았으면, 책임을 수행할 객체를 해당 타입의 인스턴스로 만들자.
- 메시지를 처리하기에 적합한 객체는?(누가 커피를 주문할 책임을 져야 하는가)
    - 당연히 손님
    - 따라서 메시지를 처리할 객체는 손님 타입의 인스턴스이다.
- 손님이 커피를 주문하는 도중에 스스로 할 수 있는 일이 무엇이 있을지 생각해보자
    - 스스로 할 수 없는 일이 발견된다면, 이는 다른 객체에게 요청해야 할 사항이다.
    - 여기서 손님 객체에서 외부로 전송되는 메시지를 정의해야 한다.
- 손님은 메뉴 항목에 대해 알지 못한다.
    - 여기서 메뉴 항목을 찾아라 라는 새로운 메시지
- 메뉴 항목을 찾아라 라는 메시지는 누구에게 할당하는 것이 좋을까?
    - 메뉴판 객체!
    - 현실 속의 메뉴판 객체는 스스로 메뉴 항목을 찾지 않지만, 소프트웨어 세계에서는 이렇게 하는게 자연스레 더 좋음을 느낄 수 있다!
- 손님은 이제 메뉴판 객체의 회신을 받고(커피 메뉴를 받음) 해당 메뉴 항목에 맞는 커피를 제조해 달라고 요청할 수 있다.
    - 새로운 요청은 새로운 메시지가 필요하다는 신호!
- 누가 커피를 제조해야 하는가?
    - 당연히 바리스타~
- 바리스타는 커피 고수이기 때문에, 어떤 커피가 요청으로 들어오던 바리스타는 어쨋건 알아서 다 만들어 낼 것이다.
    - 즉 상세한 방법은 전달할 필요가 없다는 것!

### 인터페이스 정리하기

- 위 일련의 과정을 통해 얻어낸 것은 결국 객체들의 인터페이스이다!
    - 객체가 수신한 메시지가 객체의 인터페이스를 결정한다…
    - 메시지가 객체를 선택했고, 선택된 객체는 메시지를 자신의 인터페이스로 받아들인다!
- 객체가 어떤 메시지를 수신할 수 있다는 것은…
    - 그 객체의 인터페이스 안에 메시지에 해당하는 오퍼레이션이 들어있다는 것
    - ex>
        - 손님 객체의 인터페이스에는 커피를 주문하라
        - 메뉴판 객체의 인터페이스에는 메뉴 항목을 찾아라
        - 바리스타 객체의 인터페이스에는 커피를 제조하라
        - 등등…
- 객체의 타입을 구현하는 일반적인 방법:
    - 클래스를 이용하는 것
- 결과적으로 추출된 인터페이스!
    
    ```java
    class Customer {
    		public void order(String menuName) {}
    }
    
    class MenuItem {
    
    }
    
    class Menu {
    		public MenuItem choose(String name) {}
    }
    
    class Baristar {
    		public Coffee makeCoffe(MenuItem menuItem) {}
    }
    
    class Coffee {
    		public Coffee(MenuItem menuItem) {}
    }
    
    class MenuItem {
    
    }
    ```
    

### 구현하기

- 클래스의 인터페이스를 식별했으므로..
    - 이제 오퍼레이션을 수행하는 방법을 메서드로 구현해야 함!
- Customer의 협력.
    - Menu에게 menuItem에 해당하는 MenuItem을 찾아달라고 요청
    - MenuItem을 받아 Baristar에게 전달하여 원하는 커피를 제조해달라고 요청
- 문제 발생
    - Customer가 어떻게 Menu객체와 Baristar객체에 접근할 것인가
        - 객체가 다른 객체에 메시지를 보내기 위해서는 결국 참조를 얻어야 하기 때문에..
    - 다음과 같은 수정 진행
    
    ```java
    class Customer {
    		public void order(String menuName, Menu menu, Baristar baristar) {}
    }
    ```
    
    - order() 메서드의 구현 채우기
    
    ```java
    class Customer {
    		public void order(String menuName, Menu menu, Baristar baristar) {
    			MenuItem menuItem = menu.choose(menuName);
    			Coffee coffee = baristar.makeCoffee(menuItem);
    			// ...
    		}
    }
    ```
    
    - 중요!: 구현 도중에 인터페이스는 언제든 변경될 수 있다.
- Menu는 menuName에 해당하는 MenuItem을 찾아야 하는 책임이 있다.
    - 내부적으로 Menu가 MenuItem을 관리하고 있어야 할 것…
    
    ```java
    class Menu {
    		private List<MenuItem> items;
    	
    		public Menu(List<MenuItem> items) {
    			this.items = items
    		}
    		
    		public MenuItem choose(String name) {
    			for(MenuItem each : items) {
    				if(each.getName().equals(name) {
    					return each
    				}
    			}
    			return null
    		}
    }
    
    ```
    
- Baristar는 MenuItem을 이용하여 커피를 제조한다
    
    ```java
    class Baristar {
    		public Coffee makeCoffe(MenuItem menuItem) {
    			Coffee coffe = new Coffee(menuItem)
    			return coffee
    		}
    }
    ```
    
- Coffee 는 자기 자신을 생성하기 위한 생성자를 제공!
    
    ```java
    class Coffee {
    		private String name;
    		private int price;
    		
    		public Coffee(MenuItem menuItem) {
    			this.name = menuItem.getName();
    			this.price = menuItem.cost();
    		}
    }
    ```
    

## 코드와 세 가지 관점

### 도메인 개념을 참조하는 이유

- 어떤 메시지가 있을 때, 그 메시지를 수신할 객체를 어떻게 선택하는가?
    - 전략
        - 도메인 개념 중에서 가장 적절한 것을 선택하는 것
        - 도메인 개념 안에서 적절한 객체를 선택하는 것은 도메인에 대한 지식을 기반으로 코드의 구조와 의미를 쉽게 유추할 수 있게 함!
            - 이는 시스템의 유지보수에 큰 영향

### 인터페이스와 구현을 분리하라

- 다시 한번 강조한다. 인터페이스와 구현을 분리하라
- 명세 관점은…
    - 클래스의 안정적인 측면을 드러내야 한다.
- 구현 관점은…
    - 클래스의 불안정한 측면을 드러내야 한다.
- 즉 인터페이스가 구현 세부 사항을 노출하기 시작하면 아주 작은 변동에도 전체 협력이 요동치는 취약한 설계가 된다…
- 물론 개념적인 관점과 명세적인 관점 사이는 그렇게 중요하지 않은  경우가 많다고 한다.
- 중요한 것은 클래스를 봤을 때, 명세 관점과 구현 관점으로 나눠볼 수 있어야 한다는 것…
