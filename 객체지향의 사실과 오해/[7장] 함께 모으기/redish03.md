## 클래스를 바라보는 세 가지 관점
설계와 구현의 과정을 설명하기에 앞서, 저자는 마틴 파울러(Martin Fowler)가 제시한 객체지향 설계를 바라보는 **세 가지 관점(Perspective)**을 소개했다. 훌륭한 객체지향 코드는 하나의 클래스 안에 이 세 가지 관점을 모두 명확하게 담아낸다.

개념 관점 (Conceptual Perspective): 사용자가 도메인(문제 영역)을 바라보는 관점을 반영한다. 이 관점에서는 실제 도메인의 규칙과 제약을 최대한 유사하게 반영하여 클래스를 도출한다. 즉, "이 소프트웨어가 해결하려는 현실 세계의 핵심 개념은 무엇인가?"를 다룬다.

명세 관점 (Specification Perspective): 도메인의 개념을 벗어나, 개발자의 영역으로 들어오는 관점이다. 이 관점에서는 객체가 협력을 위해 '무엇'을 할 수 있는지, 즉 객체의 **공용 인터페이스(Public Interface)**에 초점을 맞춘다. 객체가 수신할 수 있는 메시지를 결정하는 것이 핵심이다.

구현 관점 (Implementation Perspective): 객체들이 책임을 수행하기 위해 내부적으로 '어떻게' 동작할 것인지를 다룬다. 객체의 상태(속성)를 어떻게 저장하고, 메서드(행동)를 어떻게 구현할 것인지 결정한다. 철저하게 클래스의 내부 구현에 초점을 맞춘다.

이 세 가지 관점은 폭포수 모델처럼 순차적으로 일어나는 단계가 아니다. 코드를 작성할 때 개발자는 이 세 가지 관점을 자유롭게 오가며 클래스를 설계하고 다듬어야 한다.

### 1단계: 커피 전문점 도메인 모델링 (개념 관점)
소프트웨어 설계의 첫 단추는 도메인 모델을 구축하는 것이다. 커피 전문점이라는 도메인에서 핵심적인 개념(객체)들과 그들의 관계를 파악한다.

객체 도출: 커피 전문점에는 '손님', '메뉴판', '메뉴 항목(아메리카노, 카푸치노 등)', '바리스타', '커피'라는 개념이 존재한다.

관계 설정: 손님은 메뉴판을 알아야 하고, 메뉴판은 여러 메뉴 항목을 포함한다. 손님은 바리스타를 알아야 커피를 주문할 수 있으며, 바리스타는 커피를 제조한다.

이렇게 정리된 도메인 모델은 객체들의 윤곽을 잡는 훌륭한 출발점이 된다. 소프트웨어의 객체는 이 도메인 모델에 나타난 개념들을 '은유'하여 창조될 것이다.

### 2단계: 메시지와 책임 찾기 (명세 관점)
도메인 모델의 뼈대를 잡았다면, 이제 객체들이 어떻게 협력할지 설계할 차례다. 객체지향 설계의 핵심은 '객체'가 아니라 **'메시지'**를 먼저 선택하는 것이다.

메시지가 객체를 선택하게 해야 한다.

첫 번째 메시지 찾기: 커피 전문점 시스템의 가장 큰 책임은 '커피를 주문하는 것'이다. 따라서 첫 번째 메시지는 **"커피를 주문하라"**가 된다.

메시지 수신자 선택: "커피를 주문하라"는 메시지를 처리할 책임을 누구에게 할당할 것인가? 도메인 모델에서 가장 적합한 후보는 **'손님(Customer)'**이다.

협력 요청 (새로운 메시지 도출): 손님이 스스로 커피를 주문하려면 자신이 원하는 메뉴를 찾아야 한다. 하지만 손님은 메뉴의 상세 정보를 모른다. 따라서 손님은 메뉴 정보를 알고 있는 객체에게 도움을 요청해야 한다. 여기서 **"메뉴 항목을 찾아라"**라는 새로운 메시지가 탄생한다.

다음 수신자 선택: 메뉴 항목을 찾을 책임을 질 객체는 누구인가? 메뉴 정보를 포함하고 있는 **'메뉴판(MenuBoard)'**이다. 메뉴판은 내부에서 해당하는 '메뉴 항목(MenuItem)'을 찾아 손님에게 반환한다.

다시 협력 요청: 원하는 메뉴 항목을 얻은 손님은 이제 진짜로 커피를 만들어 달라고 요청해야 한다. **"커피를 제조하라"**라는 메시지를 던진다.

마지막 수신자 선택: 커피를 제조할 지식과 기술을 가진 객체는 **'바리스타(Barista)'**다. 바리스타는 전달받은 메뉴 항목을 바탕으로 '커피(Coffee)' 객체를 생성하여 손님에게 반환함으로써 전체 협력이 완성된다.

이 과정을 통해 객체들의 공용 인터페이스(수신 가능한 메시지의 목록)가 명확하게 정의된다. 손님은 order(), 메뉴판은 choose(), 바리스타는 makeCoffee()라는 명세를 갖게 된다.

### 3단계: 인터페이스와 구현의 분리 (구현 관점)
명세 관점을 통해 객체의 윤곽(인터페이스)이 잡혔다면, 이제 내부를 채워 넣는 '구현' 단계로 넘어간다.

상태 결정: 객체가 책임을 수행하기 위해 어떤 정보(상태)를 알아야 하는지 결정한다. 예를 들어 메뉴판 객체는 내부에 메뉴 항목들의 목록을 List나 Map 형태로 가지고 있어야 한다.

메서드 구현: 수신한 메시지를 처리하기 위한 구체적인 로직을 작성한다. 바리스타 객체라면 넘겨받은 메뉴 항목의 이름과 가격을 바탕으로 새로운 커피 인스턴스를 생성하는 new Coffee(...) 로직을 구현하게 된다.

이 과정에서 가장 중요한 원칙은 **인터페이스와 구현의 철저한 분리(캡슐화)**다. 객체의 내부 구현(어떤 자료구조를 쓰는지, 알고리즘이 무엇인지)이 변경되더라도, 외부로 노출된 공용 인터페이스(메시지)가 변하지 않는 한 협력하는 다른 객체들에게는 아무런 영향을 미치지 않아야 한다.

## 결론: 코드에 숨겨진 세 가지 관점
저자는 7장을 마무리하며, 완성된 코드를 다시 한번 되짚어보라고 권한다. 잘 짜인 객체지향 코드는 하나의 클래스 파일 안에서 앞서 말한 세 가지 관점을 모두 찾아볼 수 있어야 한다.

클래스의 이름과 구조를 보면 도메인의 현실 개념이 떠올라야 한다. (개념 관점)

클래스의 public 메서드들만 모아보면 이 객체가 외부와 어떻게 협력하는지, 어떤 책임을 지는지 명확히 보여야 한다. (명세 관점)

클래스의 private 속성과 메서드 내부 로직을 보면 외부의 간섭 없이 객체 스스로 상태를 관리하고 책임을 자율적으로 완수하는 방식이 드러나야 한다. (구현 관점)

객체지향 패러다임이란 현실을 단순 모방하는 것이 아니라, 명확한 책임을 가진 객체들이 메시지라는 인터페이스를 통해 소통하며 유연하고 변경 가능한 시스템(코드)을 구축해 나가는 실용적인 기술이라는 점이다.
