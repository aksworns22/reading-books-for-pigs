> 객체지향 패러다임은 지식을 추상화하고 추상화한 지식을 객체 안에 캡슐화 함으로써 실세계 문제에 내재된 복잡성을 관리하려고 한다. 객체를 발견하고 참조하는 것은 지식과 행동을 구조화 하려는 문제이다. - 레베카 워프스브록
> 

### 객체지향과 인지 능력

인간은 세상을 독립적이고 식별 가능한 객체의 집합으로 본다. 다른 사람들이 객체지향을 직관적이고 이해하기 쉬운 패러다임이라고 하는 이유다.

객체는 물리적 경계를 지닌 사물일 수 도 있지만, 인간의 인지 능력은 개념적인, 추상적인 사물과 개념도 가능하다. 예를 들면, 계산이나 주문, 계좌 이체와 같은 것들은 물리적 실체는 없지만 인간이 쉽게 구분하고 하나의 단위로 인지하는 개념적 객체이다.

세상을 더 작은 객체로 분해하는 것은 이해를 위한 몸부림이다. 

OOP는 인간의 세계에서 객체들이 모여 현실을 이루는 것처럼 SW도 그럴 것이라는 생각에서 시작한다.

그렇지만, 저기까지다. OOP의 목적은 현실 세계를 모방하는 것이 아닌, 이를 기반으로 새로운 세계를 창조하는 것이다. 따라서 SW의 객체는 현실에 존재하는 객체와 전혀 다른 모습을 띈다. 현실은 사람이 직접 주문 금액을 계산하지만, SW세계에서는 주문 객체가 자신의 금액을 계산한다.

### 객체, 그리고 이상한 나라

이상한 나라의 앨리스 예시를 든다. 토끼를 쫓다가 정원의 문을 발견했는데, 본인의 키가 너무 커서 못들어가다가 키 작아지는 물약을 먹고 작아지고, 반대로 열쇠를 잡을 수 없어 케이크를 먹고 커지고, 버섯을 먹어 키를 조절해 문을 통과하게 되는 이야기였다.

**앨리스 객체**

앨리스는 정원으로 통하는 문을 통과하기에 적당한 상태로 자신의 키를 계속해서 변화시켰다. 문을 통과하기에 적당한 상태란 문을 통과하기에 적절한 키이고, 특정 시점의 앨리스의 상태란 특정 시점에서의 앨리스의 키이다.

앨리스의 키를 변화 시킨 것은 행동이다. 케이크를 먹는 것, 버섯을 먹는 것, 부채질을 하는 것 등. 앨리스의 상태를 결정 하는 것은 행동이지만, 행동의 결과를 나타내는 것은 상태이다. 행동을 하기 전 앨리스의 키가 얼마였냐에 따라 행동을 한 후의 앨리스의 키를 결정한다. 즉, 상태에 의존적이다.

앨리스는 상태가 바뀌었지만, 주변 사람들 모두가 앨리스를 앨리스라고 부른다. 앨리스는 상태 변경과 무관하게 유일한 존재로 식별 가능하다.

1. 앨리스는 상태를 가지며, 상태는 변경 가능하다
2. 앨리스의 상태를 변경시키는 것은 앨리스의 행동이다
    1. 행동의 결과는 상태에 의존적이며 상태를 이용해 서술할 수 있다.
    2. 행동의 순서가 결과에 영향을 미친다.
3. 앨리스는 어떤 상태에 있더라도 유일하게 식별 가능하다.

### 객체, 그리고 소프트웨어 나라

> 객체의 정의
——————
이 책에서는 객체란 식별 가능한 개체 또는 사물이다. 객체는 구체적 사물일수도, 추상적 개념일수도 있다. 객체는 구별 가능한 식별자, 특징적인 행동, 변경 가능한 상태를 가진다. SW안에서 객체는 저장된 상태와 실행 가능한 코드를 통해 구현된다.
> 

**상태**

- 왜 상태가 필요한가

모든 일의 공통점은, 어떤 행동의 결과는 과거에 어떤 행동들이 일어냤느냐에 의존한다. 일반적으로 과거에 발생한 행동의 이력을 통해 현재 발생한 행동의 결과를 판단하는 것은 번거롭고 이해가 어렵다. 따라서 인간은 행동의 과정과 결과를 단순하게 기술하기 위해 상태란 개념을 만들었다.

- 예시
    - 비행기 탑승 여부는 항공권의 발권 상태를 보고 예측할 수 있다.
    - 자판기는 현재까지 투입된 금액의 상태를 기억한다.
    - TV 전원이 들어온 상태여야만 채널을 변경할 수 있다.
    - 여자친구와 즐거운 데이트를 즐기려면 자동차의 시동이 걸려있는 상태여야한다.

상태를 이용하면 과거의 모든 행동 이력을 설명없이 행동의 결과를 쉽게 예측 가능하고 설명할 수 있다. 앨리스가 과거에 뭘 했든, 키만 알면 문의 통과 여부를 알 수 있다. 과거에 얽매이지 않고, 현재를 기반으로 객체의 행동 방식을 이해할 수 있다. 상태를 통해 근본적으로 세상의 복잡성을 완화하고 인지 과부하를 줄일 수 있다.

- 상태와 프로퍼티

앨리스, 토끼, 케이크, 부채, 버섯, 문 모두 객체다. 숫자, 문자열, 양, 속도, 시간, 날짜, 참 거짓과 같은 단순 값들은 객체가 아니다. 이들은 객체의 상태를 표현하기 위한 중요한 수단이다. 때론 단순 값 뿐만 아니라 객체를 사용해 다른 객체의 상태를 표현해야 한다. 만약 앨리스가 0.5L 음료라는 객체를 들고 있다고 생각할 수 있다. 그럼 앨리스는 0.5L 크기의 음료를 가지고 있다.

모든 객체의 상태는 단순한 값과 객체의 조합으로 표현할 수 있다. 객체의 상태를 구성하는 모든 특징을 통틀어 프로퍼티(property) 라고 한다. 키, 위치, 음료가 앨리스의 프로퍼티이고, 일반적으로 변경되지 않고 고정되어 ‘정적’이다. 반면 프로퍼티 값(property value)는 시간이 흐름에 따라 변화하기 때문에 ‘동적’이다.

객체와 객체 간 의미있는 연결, 즉, 앨리스와 음료의 사이의 연결을 링크(link)라고 한다. 객체 간엔 링크가 존재해야만 요청을 보내고 받을 수 있다. 만약 앨리스가 음료를 버린다면, 이 링크가 없어지고, 더 이상 협력이 불가능하다. 

이와 달리 단순한 값은 속성(Attribute) 라고 한다. 키, 위치는 단순한 값으로 표현되어 속성이다.

> 상태의 정의
——————
이 책에서는 특정 시점에 객체가 가지고 있는 정보의 집합으로 객체의 구조적 특징을 표현한다고 정의.
상태는 객체에 존재하는 정적 프로퍼티와 동적 프로퍼티 값으로 구성된다. 객체 프로퍼티는 단순한 값과 다른 객체를 참조하는 링크로 구분할 수 있다.
> 

자율적인 객체는 스스로 자신의 상태를 책임져야 한다. 외부의 객체가 직접 객체의 상태를 주무를 수 없다.

**행동**

- 상태와 행동

객체가 취하는 행동은 객체 자신의 상태를 변경시킨다. 객체의 행동에 의해 객체의 상태가 변경된다는 것은 행동이 부수 효과(side-effect)를 초래하는 것이다. 

상태와 행동 간엔 다음과 같은 관계가 있다.

1. 객체의 행동은 상태에 영향을 받는다.
2. 객체의 행동은 상태를 변경시킨다.

이는 상태를 이용해 행동을 아래 두 가지 관점에서 서술 가능하다.

1. 상호작용이 현재의 상태에 어떤 방식으로 의존하는가
2. 상호작용이 어떻게 현재의 상태를 변경시키는가
- 비유
    
    앨리스가 통과해야하는 문의 크기가 40cm라고 친다면, 문을 통과하는 행동은 과거의 행동을 볼 필요 없이 앨리스의 ‘키’와 ‘위치’ 라는 두 가지 상태를 이용해 간단히 서술할 수 있다.
    
    1. 앨리스의 키가 40 이하라면 문을 통과할 수 있다.
    2. 문을 통과한 후에 앨리스의 위치는 아름다운 정원으로 바뀌어야 한다.

다시 말하지만, 상태를 이용하면 복잡한 객체의 행동을 쉽게 이해할 수 있다.

- 협력과 행동

객체는 다른 객체와 상호작용 해 협력하는 객체들의 공동체를 만든다. 객체가 서로 메시지를 주고 받고 하는 상황에서 객체는 자기 자신 뿐만 아니라 다른 객체의 상태 변경을 유발 할 수 있다.

앨리스가 음료를 마시면, 앨리스 자신의 키가 작아지는 동시에 음료의 남아있는 양 상태 변경도 유발한다.

객체의 행동은 이 두 가지 관점의 부수효과를 명확히 설명해야한다.

1. 객체 자신의 상태 변경
2. 행동 내에서 협력하는 다른 객체에 대한 메시지 전송

> 협력의 정의
——————
이 책에서는 외부의 요청 또는 수신된 메시지에 응답하기 위해 동작하고 반응하는 활동이라고 정의.
행동의 결과로 객체는 자신의 상태를 변경하거나 다른 객체에게 메시지 전달 가능. 객체는 행동을 통해 다른 객체와의 협력에 참여하므로 행동은 외부에 가시적이어야 한다.
> 

- 상태 캡슐화

![image.png](attachment:9d7bca58-9e3b-442d-92db-81335fa8a59f:image.png)

위의 그림을 보고 앨리스의 키가 줄어드는거, 음료의 양이 줄어드는 것을 예상할 수 있을까? 알지 못한다.

이것이 캡슐화가 의미하는 것이다. 객체는 상태를 캡슐 안에 감춰둔 채 외부로 노출하지 않는다. 객체가 외부에 노출하는 것은 행동 뿐이며, 외부에서 객체에 접근할 수 있는 유일한 방법 역시 행동뿐이다.

객체의 행동을 유발하는 것은 외부에서 전달된 메시지지만, 상태를 변경할지 여부를 정하는 것은 객체 스스로이다. 사실 객체에 메시지를 던지는 외부는 수신하는 객체가 변경된다는 사실 조차 모른다. 메시지를 해석하고 반응해 상태를 변경할지 여부를 결정하는 것은 전적으로 메시지 수신자의 자율적 판단이다.

> 상태를 외부에 노출하지 않고, 행동을 경계로 캡슐화 하는 것은 결과적으로 객체의 자율성을 높인다.
> 

스스로 판단하고, 스스로 결정하기 때문에 자율성이 높을수록 지능이 높아지고, 협력은 유연해지고, 간결해진다.

**식별자**

객체를 서로 구별할 수 있는 특정한 프로퍼티가 객체 안에 존재한다. 이 프로퍼티를 식별자라고 한다. 모든 객체는 식별자를 가지며, 이를 이용해 구분 가능하다.

값은 식별자를 가지지 않지만, 객체는 가진다!

값(value)은 숫자, 문자열, 시간, 금액과 같은 변하지 않는 양을 모델링한다. 흔히 값의 상태는 변하지 않아 불변 상태를 가진다 한다. 두 개의 1이라는 숫자가 종이 위에 적혀 있다면 두 인스턴스를 같다고 판단한다. 값이 같은지 여부는 상태가 같은지를 이용해 판단한다. 상태가 같다면 동일, 아니라면 다른 것이다. 이렇게 두 값이 같은지 판단하는 성질을 동등성(Equality)라고 한다. 이게 가능한 이유는 값의 상태는 불변이라 그렇다.

객체는 시간에 따라 변경되는 상태를 포함한다. 따라서 객체는 가변 상태를 포함한다. 타입이 같은 두 객체의 상태가 완전히 같아도, 두 별개의 독립적 객체로 다뤄야한다. 어린 시절의 나와 지금의 내가 키도 다르고, 나이도 적지만 두 사람은 동일인물이다. 그렇지만 내가 완전 같은 쌍둥이를 가지고 있어도, 이름이 달라 다른 개체이다. 식별자를 기반으로 객체가 같은지 판단 하는 것을 동일성(Identical)이라고 한다.

상태는 항상 변하기 때문에, 상태를 기반으로 동일성을 판단할 수 없다.

> 식별자의 정의
——————
이 책에서는 어떤 객체를 다른 객체와 구분하는 데 사용하는 객체의 프로퍼티로 정의
값은 식별자를 가지지 않아 상태를 이용한 동등성 검사를 통해 두 인스턴스를 비교해야한다.
객체는 상태가 변경될 수 있어 식별자를 이용한 동일성 검사를 통해 두 인스턴스를 비교해야한다.
> 

대부분은 객체와 값의 차이를 혼란스러워 하는데, 둘다 클래스를 이용해 구현되기 때문이다. 숫자는 Integer, 사람은 Person이라는 클래스로 정의할 수 밖에 없다.

이런 오해를 줄이기 위해 객체와 값을 별도로 지칭하는 용어를 사용하기도 한다.

1. 참조 객체(Reference Object) 또는 엔티티(Entity) → 식별자를 지닌 전통적 의미의 객체
2. 값 객체(Value Object) → 식별자를 가지지 않는 값.

객체의 특성은 아래와 같다. 사실 이전의 앨리스에서 객체로 살짝 바꾼게 다다.

1. 객체는 상태를 가지며, 상태는 변경 가능하다
2. 객체의 상태를 변경시키는 것은 객체의 행동이다
    1. 행동의 결과는 상태에 의존적이며 상태를 이용해 서술할 수 있다.
    2. 행동의 순서가 결과에 영향을 미친다.
3. 객체는 어떤 상태에 있더라도 유일하게 식별 가능하다.

### 기계로서의 객체

쿼리: 객체의 상태를 조회하는 작업

명령: 객체의 상태를 변경하는 작업

객체는 블랙박스다. 밖의 사람은 기계 내부를 볼 수 없다. 대신 사람은 밖의 사각형과 원모양의 버튼만 이용해서 기계와 상호작용 가능하다. 

이런 비유는 캡슐화를 강조한다. 외부에 객체가 제공하는 것은 행동 뿐이다. 

객체가 상태와 무관한 구분 가능한 식별자를 지닌다는 것 또한 강조한다는 뜻이다. 어떤 두 기계가 완전히 동일한 동작을 제공하고 같은 상태를 가진다, 그렇지만 사람들은 이 두 기계를 별개의 객체로 인식한다. 

### 행동이 상태를 결정한다

초보자들이 실수하기 쉬운 건 상태를 중심으로 객체를 바라보는 것.

1. 상태를 먼저 결정하고 행동을 먼저 결정하는 방법은 설계에 나쁜 영향을 미친다.
    1. 캡슐화 저해
    2. 객체를 협력자가 아닌 고립된 섬으로 만듦
        - 상태를 먼저 고려하면 문맥은 덜 고려되기 때문에,
    3. 객체의 재사용성이 저해
        - 상태에 초점을 맞추면 다양한 협력에 참여하기 어렵다..

결론적으로 행동에 초점을 맞춰야 한다. 행동은 결국 객체가 협력에 참여하면서 완수해야 하는 책임이므로, 어떤 책임이 필요한지 결정하는 과정이 설계를 주도해야한다. 책임-주도-설계(RDD) 또한 비슷한 맥락으로, 응집도 높고 재사용 가능한 객체를 만들 수 있게 한다.

### 은유와 객체

**두 번째 도시전설**

객체지향이란 현실세계의 모방?

단순한 모방이 아니다. 소프트웨어 안의 상품은 가격 계산을 할 수 있다. 실제 상품은 못하지만, 객체는 스스로 해 내는 것이다. 모방과 추상화만으로는 현실과 SW상의 객체를 구분하지 못한다.

**의인화**

현실 속의 객체와 소프트웨어 객체 간의 가장 큰 차이는 능동적으로 변한다는 것. 이를 현실 속의 객체를 마치 의인화 했다고 할 수 있다.

**은유**

설계를 잘하는 전통적 조언은 현실 세계의 객체를 자세히 관찰하고, 그 중 SW 객체에 적합한 속성만 추리는 것이다. 다만 이는 도움이 되지 않는다.

현실 객체의 은유를 사용한다면 표현적 차이를 줄일 수 있고, 이해하기 쉽고 유지보수가 용이한 SW를 만들 수 있다. 이런 이유로 모든 OOP 지침서엔 현실 세계인 도메인에 사용되는 이름을 객체에 부여하라고 하는 것이다.

**이상한 나라를 창조하라**

객체지향 설계자의 목적은 현실 모방이 아니다.

1.  이상한 나라를 창조하자
2. 현실을 닮아야 한다는 제약, 구속은 없다.
3. 우리가 창조한 객체의 특성을 상기시킬 수 있다면 현실 속의 객체 이름을 이용해 묘사하자. 아니라면 깔끔하게 현실을 무시하고 나만의 자유로운 세계를 만들어도 된다.
