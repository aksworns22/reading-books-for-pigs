# 2. 이상한 나라의 객체

## 객체지향과 인지 능력

- 인간의 인지능력을 통해 인간은 물리적인 한계를 넘어 개념적으로 경계 지을 수 있는 추상적인 사물까지도 객체로 인식할 수 있게 해준다.
- 객체지향 패러다임이 인간이 인지할 수 있는 다양한 객체들이 모여 현실세계를 이루고자 하는 것처럼, 소프트웨어 세계역시 인간이 인지할 수 있는 다양한 소프트웨어 객체들이 모여 이뤄져 있다는 믿음에서 출발한다!
- 하지만 소프트웨어 세계의 객체와 현실세계의 객체는 전혀 다르다
    - 현실세계의 전등은 사람의 손길 없이 스스로 불을 밝힐 수 없지만, 소프트웨어 세계의 전등은 외부의 도움 없이도 스스로 전원을 켜거나 끌 수 있다!

## 객체 그리고 이상한 나라

### 이상한 나라의 앨리스

- 이상한 나라의 앨리스에서 앨리스는 작은 문을 통과하기 위해 온갖 고초를 겪는다,

### 앨리스 객체

- 문을 통과하기 위한 적절한 상태란 무엇인가?
    - 문을 통과하기에 적절한 키를 의미할 것이다.
- 앨리스의 키는 시간의 흐름에 따라 변화했다, 그렇지만 아무런 이유없이 변하지는 않았다.
    - 음료를 마시거나, 부채질을 하거나, 버섯을 먹거나, 케이크를 먹거나…..등등
    - 결국 앨리스의 키를 변화시킨 것은 앨리스의 행동이다.
- 행동은 앨리스의 상태를 결정하고, 상태는 행동의 결과를 결정한다.
    - 왜 앨리스의 행동의 결과는 앨리스의 상태에 의존하는가?
        - 케이크를 먹었을 때 키가 20cm 자란다고 하였을 때, 이것만 가지고는 행동의 결과를 알 수 없음!
        - base 키가 얼마인지 알아야 한다..
- 앨리스가 얼마나 커지던 작아지던 앨리스는 앨리스이다.

## 객체, 그리고 소프트웨어 나라

- 하나의 개별적인 실체로 식별 가능한 물리적인, 혹은 개념적인 사물은 어떤 것이라도 객체가 될 수 있다!
- 객체의 다양한 특성을 효과적으로 설명할 수 있는 방법
    - 객체를 ,상태, 행동, 식별자를 지닌 실체로 보는 것
- 이 책에서는 객체를 다음과 같이 정의하기로 한다.
    
    > 객체란 식별 가능한 개체 또는 사물이다. 객체는 자동차처럼 만질 수 있는 구체적인 사물일 수도 있고, 시간처럼 추상적인 개념일 수도 있다, 객체는 구별 가능한 식별자, 특징적인 행동, 변경 가능한 상태를 가진다. 소프트웨어 안에서 객체는 저장된 상태와 실행 가능한 코드를 통해 구현된다.
    > 

### 상태

#### 왜 상태가 필요한가?

- 객체가 주변 환경에 상호작용에 어떻게 반응하는지는 어떻게 알 수 있겠는가?
    - 그 시점까지 객체에 어떤 일이 발생했느냐에 따라 좌우된다.
        - Ex> 비행기를 이용할 때 미리 항공권을 발권해두었다면 탈 수 있지만, 그렇지 않다면 탈 수 없다.
    - 결론적으로 어떤 행동의 결과는 과거에 어떤 행동이 일어났었느냐에 의존한다는 것이다.
- 하지만 이는 조금 문제가 있을 수 있다.
    - 과거에 모든 행동을 기억해야 한다.
    - 복잡하고 번거롭다.
- 다음과 같이 생각하는 것이 좀 더 좋다.
    - 비행기 탑승 가능 여부는 항공권의 발권 상태를
    - 자판기는 현재까지 투입된 금액의 상태를
    - etc
- 결론적으로 상태를 사용하면, 과거의 모든 행동 이력을 설명하지 않고도 행동의 결과를 쉽게 예측하고 설명할 수 있다!

#### 상태와 프로퍼티

- 세상에 존재하는 모든 것이 객체인가?
    - No,
    - 분명하게 인식할 수 있음에도 객체의 영역에 포함시키기에 애매한 것들도 존재한다.
        - 앨리스의 키와 위치, 음료의 케이크의 양 등등..
- 단순한 값들은 객체가 아니다, 그렇다면 이들은 어떻게 처리해야 하는가?
    - 다른 객체의 특성을 표현하는 데에 사용할 수 있다!
    - 객체는 아닐지라도, 중요하다…
- 그렇다면 다른 객체의 특성을 표현한다 == 객체가 아님 인가?
    - No,
    - 때로는 단순한 값이 아니라 객체를 사용하여 다른 객체의 상태를 표현해야 하는 때가 존재한다.
- 앨리스가 현재 음료를 들고 있을 때, 이를 어떻게 표현할 것인가?
    - 앨리스의 상태 일부를 음료라는 객체를 이용하여 표현하기
- 그렇다면 다음을 알 수 있다
    
    > 모든 객체의 상태는 단순한 값과 객체의 조합으로 나타낼 수 있다.
    > 
- 링크: 객체와 객체 사이의 의미있는 연결
    - 객체의 링크만이 메시지를 주고받게 해줄 수 있다!
    - 다른 객체가 다른 객체를 참조할 수 있다는 의미기도 하다, 즉 식별자를 알고 있다는 것
- 이 책에서는 객체의 상태를 다음과 같이 정의한다.
    
    > 상태는 특정 시점에 객체가 가지고 있는 정보의 집합으로 객체의 구조적 특징을 표현한다. 객체의 상태는 객체에 존재하는 정적인 프로퍼티와 동적인 프로퍼티 값으로 구성된다. 객체의 프로퍼티는 단순한 값과 다른 객체를 참조하는 링크로 구분할 수 있다.
    > 
- 객체는 자율적이다 즉
    - 객체는 다른 객체의 상태에 직접적으로 접근할 수도, 상태를 변경할 수도 없다.

### 행동

#### 상태와 행동

- 객체의 상태는 언제 변경되는가?
    - 객체의 상태는 객체의 자발적인 행동에 의해서만 변경된다.
        - ex> 앨리스가 음료를 마셔서 키가 작아졌다
        - ex> 앨리스가 문을 통과해서 위치가 통로에서 정원으로 바뀌었다.
- 부수효과: 객체가 취하는 행동에 의해 상태가 변경된다.
    - ex> 앨리스가 케이크를 먹는 행위는 다음과 같은 부수효과를 발생시킨다.
        - 키를 작게 변화시킨다.
        - 케이크의 양을 줄인다.
- 상태와 행동 사이의 관계
    - 객체의 행동은 상태에 영향을 미친다
    - 객체의 행동은 상태를 변경시킨다.
- 행동을 상태라는 개념을 이용하여 다음과 같이 서술할 수 있다.
    - 상호작용이 현재의 상태에 어떤 방식으로 의존하는가
    - 상호작용이 어떻게 현재의 상태를 변경시키는가

#### 협력과 행동

- 요청: 객체가 다른 객체와 협력하기 위한 유일한 방법
- 객체가 다른 객체와 의사소통 하는 방법은 메시지이므로, 다음과 같이 생각해볼 수 있다.
    - 객체는 수신된 메시지에 따라 적절히 행동하면서 협력에 참여하고, 그 결과로 자신의 상태를 변경한다.
- 객체의 행동은 다음과 같은 부수효과를 발생시킨다.
    - 객체 자신의 상태 변경
    - 행동 내에서 협력하는 다른 객체에 대한 메시지 전송
- 이 책에서는 행동을 다음과 같이 정의한다.
    
    > 행동이란 외부의 요청 또는 수신된 메시지에 응답하기 위해 동작하고 반응하는 활동이다. 행동의 결과로 객체는 자신의 상태를 변경하거나 다른 객체에 메시지를 전달할 수 있다.  객체는 행동을 통해 다른 객체와의 협력에 참여하므로 행동은 외부에 가시적이어야 한다.
    > 

#### 상태 캡슐화

- 기묘한 객체지향의 음료수
- 현실 세계에서 앨리스가 음료수를 마시면, 앨리스가 음료의 양을 줄이는 것처럼 보이지만, 객체 지향의 세계에서는 모든 객체는 자신의 상태를 스스로 관리해야 하므로 음료 객체의 양을 줄이는 것은 음료 자신이어야 한다.
- 앨리스 객체는 단지 음료수 객체에게 메시지를 전달할 뿐이다.
- 상태를 외부에 노출시키지 않을수록 객체의 자율성을 높인다.
    - 자율성이 높은 객체는 지능이 높다고 할 수 있다.
    - 협력에 참여하는 객체들의 지능이 높아질수록 협력은 유연하고 간결해진다.

        > 

### 식별자

- 객체가 식별 가능하다 라는 것은 무엇인가?
    - 객체를 서로 구별할 수 있는 특정한 프로퍼티가 객체 안에 존재한다는 것이다.
- 모든 객체는 식별자를 가진다, 그렇다면? 객체가 아닌 단순한 값들은 식별자를 가지지 않는다.

## 기계로서의 객체

- 객체를 기계에 빗대어 설명할 수 있다!
    - 쿼리: 객체의 상태를 조회하는 작업
    - 명령: 객체의 상태를 변경하는 작업
- 기계로서의 객체는 블랙박스이다. 즉 단단한 금속외피 안에 부품들이 있기 때문에 밖에서는 오직 버튼만을 눌러 기계와 상호작용할 수 있다.
    - 즉 캡슐화를 강조하는 비유
    - 식별자도 효과적으로 설명할 수 있다!
- 어떤 두 기계가 완전히 동일한 동작을 제공하고 같은 상태를 가진다, 그렇지만 사람들은 이 두 기계를 별개의 객체로 인식한다!
    - 객체가 상태와 무관한 구분 가능한 식별자를 지닌다는 것!

## 행동이 상태를 결정한다

- 함정: 상태를 중심으로 객체를 바라보는 것
- 상태를 먼저 결정하고 행동을 먼저 결정하는 방법은 설계에 나쁜 영향을 미친다.
    1. 캡슐화 저해
    2. 객체를 협력자가 아닌 고립된 섬으로 만듦
        - 상태를 먼저 고려하면 문맥은 덜 고려되기 때문에,
    3. 객체의 재사용성이 저해
        - 상태에 초점을 맞추면 다양한 협력에 참여하기 어렵다..
- 결론적으로 행동에 초점을 맞춰야 한다.
- 행동은 결국 객체가 협력에 참여하면서 완수해야하는 책임이므로, 어떤 책임이 필요한가? 를 결정하는 과정이 전체 설계를 주도해야 한다!
    - 책임-주도-설계(RDD)

## 은유와 객체

### 두 번째 도시전설

- 두 번째 도시전설: 객체지향이란 현실세계의 모방
    - 소프트웨어안에 구현된 상품 객체는 실제 세계의 상품과 얼마나 닮아있겠는가?
        - 소프트웨어 안의 상품은 가격 계산을 할 수 있다!
    - 즉 실제 세계의 객체를 단순화 하거나 추상화 한 것이 소프트웨어의 객체라고 할 수는 없다.

### 의인화

- 그렇다면 지금까지의 예시를 통해 다음을 유추해볼 수 있다.
    - 소프트웨어 객체는 현실 속의 객체보다 좀 더 많은 것을 할 수 있다.
    - 현실 속의 객체와 소프트웨어 객체 간의 가장 큰 차이는 능동적으로 변한다는 것.
    - 이를 현실 속의 객체를 마치 의인화 했다, 라고 할 수 있을 것이다.

### 은유

- 현실 세계의 객체를 은유를 통해 객체 지향 세계의 객체와 연관짓자!
- 은유: 실제로는 적용되지 않는 한 가지 개념을 이용하여 다른 개념을 서술하는 대화의 한 형태

### 이상한 나라를 창조하라

- 객체지향 설계자로서 우리의 목적은 다음과 같다!
    - 현실을 모방하는 것이 아니라, 이상한 나라를 창조하자
    - 현실을 닮아야 한다는 구속은 불필요하다
    - 우리가 창조한 객체의 특성을 상기시킬 수 있다면 현실 속의 객체 이름을 차용하자, 아니라면 깔끔하게 현실을 무시하고 나만의 객체를 만들어도 될 것이다.
