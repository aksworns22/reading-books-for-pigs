# 설계 품질과 트레이드오프

설계는 변경을 위해 존해, 변경에는 어떤 방식으로든 비용이 발생함 → 좋은 설계란 합리적 비용 안에서 변경을 수용할 수 있는 구조로 만드는 것

### 캡슐화

- 구현 → 변경될 가능성이 높은 부분
- 인터페이스 → 상대적으로 안정적인 부분
- 변경 가능성이 높은 부분 → 객체 내부로 숨기는 추상화 이용

### 응집도와 결합도

응집도 → 모듈에 포함된 내부 요소들이 연관돼 있는 정도 

- 모듈 내 요소들이 하나의 목적을 위해 긴밀하게 협력하면 높은 응집도
- 변경이 발샐할 때 모듈 내부에서 발생하는 변경의 정도로 측정
    
    → 하나 변경 수행 시 모듈 전체가 변경되면 높은 것 (다른 모듈이 변경되면 낮은 것)
    

결합도 → 의존성 정도, 다른 모듈에 대해 얼마나 많은 지식을 갖고 있는지

- 어떤 모듈이 다른 모듈에 대해 너무 자세한 부분까지 알고 있다면 높은 결합도
- 한 모듈이 변경되기 위해서 다른 모듈의 변경을 요구하는 정도로 측정 (다른 모듈이 얼마나 많이 변경되나?)
    
    → 하나의 모듈 변경에 다른 모듈이 많이 변경될수록 높은 것
    

변경될 확률이 매우 적은 안정적인 모듈 → 결합도가 높아도 괜찮다

### 캡슐화를 지켜라

데이터 중심의 설계 → 낮은 응집도와 높은 결합도라는 문제

→ 근본적인 원인 : 캡슐화의 원칙을 위반했기 때문

객체는 어떤 데이터를 가지고 있는지 내부에 캡슐화하고 외부에 공개하면 안된다. (인터페이스에 정의된 메서드를 통해서만 접근)

→ private로 가시성을 설정했다고 해도 접근자, 수정자를 제공한다면 캡슐화 위반

### 스스로 자신의 객체를 책임지는 객체

- 이 객체가 어떤 데이터를 포함해야하는가를 두 개의 질문으로 분할
    - 이 객체가 어떤 데이터를 포함해야 하는가?
    - 이 객체가 데이터에 대해 수행해야 하는 오퍼레이션은 무엇인가?

→ 두 개의 질문을 조합하면 객체의 내부 상태를 저장하는 방식과 저장된 상태에 대해 호출할 수 있는 오퍼레이션의 집합을 얻을 수 있음

### 데이터 중심 설계는 객체의 행동보다는 상태에 초점을 맞춘다

- 설계를 시작하는 처음부터 데이터에 관해 결정하도록 강요 → 너무 이른 시기에 내부 구현에 초점
- 데이터 중심 관점에서 객체는 그저 단순한 데이터의 집합체 → 접근자, 수정자 과도하게 추가 (캡슐화 붕괴)

### 데이터 중심 설계는 객체를 고립시킨 채 오퍼레이션을 정의하도록 만든다

- 객체지향 애플리케이션을 구현한다는 것은 협력하는 객체들의 공동체를 구축한다는 것 → 협력이라는 문맥 안에서 필요한 책임을 결정하고 이를 수행할 적절한 객체를 결정하는 것이 중요
