# 4장: 설계 품질과 트레이드오프

## 1. 데이터 중심의 영화 예매 시스템

- 객체지향 설계에서 두 가지 방법을 사용하여 시스템을 객체로 분할할 수 있다.
    1. 상태를 분할의 중심축으로 삼는 방법
    2. 책임을 분할의 중심축으로 삼는 방법
- 어느 것을 선택해야 할까?
    - 훌륭한 객체 지향 설계는 데이터(상태)가 아니라 책임을 분할의 중심축으로 삼아야 한다.
    - 객체의 상태 → 구현에 속한다, 구현은 불안정하다 → 변하기 쉽다.
    - 때문에 상태를 객체 분할의 중심으로 삼게 되면 구현에 관한 세부사항이 객체의 인터페이스에 스며들게 되어 캡슐화의 원칙이 무너지게 된다.
    - 하지만 객체의 책임은 인터페이스이기 때문에 구현 변경에 대한 파장을 막을 수 있다.

## 2. 설계 트레이드 오프

### 캡슐화

- 상태와 행동을 하나의 객체 안에 모으는 이유는 무엇인가?
    - 객체의 내부 구현을 외부로부터 감춘다.
    - 구현: 나중에 변경될 가능성이 높은 어떤 것
    - 인터페이스: 상대적으로 안정적인 부분
- 객체지향이 강력한 이유는 한 곳에서 일어난 변경이 전체 시스템에 영향을 끼치지 않도록 조절할 수 있기 때문에…
- 객체지향에서 가장 중요한 원리 → 캡슐화
- 캡슐화 → 변경 가능성이 높은 부분을 객체 내부로 숨기는 추상화 기법

### 응집도와 결합도

- 응집도와 결합도는 구조적 설계 방식이 주도하던 시기에 출범한 계획이지만 여전히 유효하다(ㄷㄷ)
- 응집도: 모듈에 포함된 내부 요소들이 연관되어 있는 정도
    - 응집도가 높다: 모듈 내의 요소들이 하나의 목적을 위해 긴밀하게 협력한다.
- 결합도: 의존성의 정도를 나타내며 다른 모듈에 대해 얼마나 많은 지식을 갖고 있는지?
    - 결합도가 높다: 어떤 모듈이 다른 모듈에 대해 꼭 필요한 지식만 알고 있지 않다.
- 그런데 이러한 요소는 꽤나 추상적이다(ㅇㅈ)
    - 어떤 요소가 얼마나 강하게 연관되어 있어야 응집도가 높다고 말할 수 있는가?
    - 모듈 사이의 어느 정도의 의존성은 남겨야 하는데, 어느정도를 남겨야 낮은 의존성이라고 할 수 있을까?
- 결합도가 높아도 상관 없는 경우가 있음
    - 변경될 확률이 매우 적은 안정적인 모듈이라면?
        - 전혀 문제가 되지 않는다.
- 캡슐화의 정도가 객체의 응집도와 결합도를 결정한다.
    - 데이터 중심의 설계는 캡슐화를 위반하기 쉽다 → 응집도가 낮고 결합도가 높은 객체들이 양산될 수 있다.

## 4. 자율적인 객체를 향해

### 캡슐화를 지켜라

- 설계의 제 1원리 → 캡슐화
    - 객체는 자신이 어떤 데이터를 가지고 있는지 내부에 캡슐화 하고 외부에 공개해서는 안된다.
    - 객체는 스스로의 상태를 책임져야 하며, 외부에서는 인터페이스에 정의된 메서드를 통해서만 상태에 접근할 수 있어야 함
- 주의: 메서드는 단순히 속성하나의 값을 반환, 변경하는 게터 세터가 아님…
    - 객체가 책임져야 하는 무언가를 수행하는 메서드
- 캡슐화 위반 사례: 속성의 가시성을 private로 설정했다고 하더라도, 게터 세터를 통해 속성을 외부로 제공하는 경우…

### 스스로 자신의 객체를 책임지는 객체

- 왜 상태와 행동을 하나의 객체로 묶는가?
    - 객체 스스로 자신의 상태를 처리할 수 있게 하기 위해.
- 이 객체가 어떤 데이터를 포함해야 하는가? 는 다음과 같은 두개의 질문으로 분리할 필요가 있다.
    - 이 객체가 어떤 데이터를 포함해야 하는가?
        - 어떤 데이터를 관리해야 하는가?
    - 이 객체가 데이터에 대해 수행해야 하는 오퍼레이션은 무엇인가?
        - 이 데이터에 대해 수행할 수 있는 오퍼레이션이 무엇인가?

## 6. 데이터 중심 설계의 문제점

- 데이터 중심 설계가 변경에 취약한 이유
    - 데이터 중심의 설계는 본질적으로 너무 이른 시기에 데이터에 관해 결정하도록 강요한다.
    - 데이터 중심의 설계에서는 협력이라는 문맥을 고려하지 않고 객체를 고립시킨 채 오퍼레이션을 결정한다.

### 데이터 중심 설계는 객체의 행동보다는 상태에 초점을 맞춘다.

- 데이터는 구현의 일부이다.
- 데이터 주도 설계는 설계를 시작하는 처음부터 데이터에 관해 결정하도록 강요하기 때문에, 너무 이른 시기에 내부 구현에 초점을 맞추게 된다.
- 데이터 중심의 관점에서 객체
    - 단순한 데이터의 집합
    - 때문에 게터세터를 과도하게 추가함
    - 데이터 객체를 사용하는 절차를 분리된 별도의 객체에서 구현하게 된다…
    - 이럴바엔 public 속성으로 선언해라..

### 데이터 중심 설계는 객체를 고립시킨 채 오퍼레이션을 정의하도록 만든다.

- 객체지향 애플리케이션을 구현한다는 것 → 협력하는 객체들의 공동체를 구축한다는 것을 의미
- 무엇이 중요한가?
    - 협력이라는 문맥 안에서 필요한 책임을 결정하고 이를 수행할 적절한 객체를 결정하는 것이 가장 중요
- 데이터 중심 설계에서는 객체의 외부가 아니라 내부에 초점을 맞춘다..
    - 객체가 관리할 데이터의 세부 정보를 먼저 결정하기 때문에, 이미 구현된 객체의 인터페이스를 억지로 끼워맞추게 된다…
