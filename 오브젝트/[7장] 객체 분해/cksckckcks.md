# 객체 분해

### 메인 함수로서의 시스템

- 하향식 접근법
    - 전통적인 기능 분해 방법
    - 시스템을 구성하는 가장 최상위 기능을 정의하고, 이 최상위 깆능을 좀 더 작은 단계의 하위 기능으로 분해해 나가는 방법
    - 분해는 세분화된 마지막 하위 기능이 구현 가능한 수준이 될 때까지 계속
- 하향식 접근법 문제점
    - 시스템은 하나의 메인함수가 아님
    - 기능 추가나 요구사항 변경으로 메인 함수를 빈번하게 수정하여야 함
    - 비즈니스 로직이 사용자 인터페이스와 강하게 결합
    - 하향식 분해는 너무 이른 시기에 함수들의 실행ㅅ 순서를 고정 → 유연성과 재사용성 저하
    - 데이터 형식이 변경될 경우 파급효과 예측 불가

### 언제 하향식 분해가 유용한가?

- 설계가 어느정도 안정화된 후에 설계의 다양한 측면을 논리적으로 설명하고 문서화할 때

### 정보 은닉과 모듈

- 정보 은닉 : 시스템을 모듈 단위로 분해하기 위한 기본 원리로 시스템에서 자주 변경되는 부분을 상대적으로 덜 변경되는 안정적인 인터페이스 뒤로 감춰야 한다는 것
- 모듈이 감춰야하는 비밀
    - 복잡성 : 모듈이 너무 복잡한 경우 이해, 사용이 어려움 → 인터페이스 제공으로 복잡도를 낮춤
    - 변경 가능성 : 변경 가능한 설계 결정이 외부에 노출될 경우 실제로 변경이 발생했을 때 파급 효과가 커짐 → 쉽게 변경되지 않을 인터페이스를 제공

### 모듈의 장점과 한계

- 장점
    - 모듈 내부의 변수가 변경되더라도 모듈 내부에만 영향을 미친다.
    - 비즈니스 로직과 사용자 인터페이스에 대한 관심사를 분리한다.
    - 전역 변수와 전역 함수를 제거함으로써 네임스페이스 오염을 방지한다.

→ 모듈이 프로시저 추상화보다는 높은 추상화 개념을 제공하지만 태생적으로 변경을 관리하기 위한 구현 기법이기 때문에 추상화 관점에서의 한계점이 명확

→ 인스턴스 개념이 없다는 단점

### 클래스는 추상 데이터 타입인가?

- 클래스는 상속과 다형성을 지원, 추상 데이터 타입은 지원 X → 클래스는 추상 데이터 타입이 아니다.
- 추상 데이터 타입 기반의 프로그래밍 패러다임 : 객체기반 프로그래밍

### 변경을 기준으로 선택하라

- 단순히 클래스를 구현 단위로 사용 → 객체지향 프로그래밍 X
- 타입을 기준으로 절차를 추상화 X → 객체지향 분해 X
- 클래스가 추상 데이터 타입의 개념을 따르는지 확인할 수 있는 가장 간단한 방법
    - 클래스 내부에 인스턴스 타입을 표현하는 변수가 있는지 살펴보기
