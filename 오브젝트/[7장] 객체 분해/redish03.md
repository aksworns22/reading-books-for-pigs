인지과부하 → 단기기억에서 문제를 해결하기 위해 필요한 요소의 수가 단기 기억의 용량을 넘어서서 문제 해결 능력이 급격하게 떨어지는 것

이를 해결하기 위해불필요한 정보를 제거하고 현재 문제해결의 핵심만 남기기 → 추상화

분해

추상화의 일반적인 방법은 큰 문제를 해결 가능한 작은 문제로 나누는 것

한 번에 단기 기억에 담을 수 있는 추상화의 수엔 한계가 있지만 추상화를 더 큰 추상화로 압축해 단기 기억의 한계를 초월할 수 있다.

### 01 프로시저 추상화와 데이터 추상화

모든 프로그래밍 패러다임은 추상화와 분해의 관점에서 설명 가능하다.

현대 프로그래밍 언어를 특정 짓는 두 가지 추상화 메커니즘은 프로시저 추상화(procedure abstraction)와 데이터 추상화(data abstraction)이다.

- 프로시저 추상화 → 소프트웨어가 무엇을 해야하는지 추상화
- 데이터 추상화 → 소프트웨어가 무엇을 알아야 하는지 추상화

현대의 설계 방법에 중요한 영향을 끼치는 프로그래밍 패러다임들은 프로시저 추상화나 데이터 추상화를 중심으로 시스템의 분해 방법을 설명한다.

시스템을 분해하는 방법을 결정하기

- 데이터 추상화를 중심으로 할 것인가, 프로시저 추상화를 중심으로 하는가?를 결정
1. 프로시저 추상화 중심 → 기능 분해(functional decomposition)의 길로 들어선다. 알고리즘 분해(Algorithm deposition)라고도 한다.
2. 데이터 추상화 중심으로 하면 아래 둘 중 하나를 선택해야한다.
    1. 타입 추상화(type abstraction) = 추상 데이터 타입(Abstract Data Type)
    2. 프로시저를 추상화(procedure abstraction) = 객체 지향(Objecet-Oriented)

지금까지 객체지향 패러다임을 역할과 책임을 수행하는 자율적인 객체들의 협력 공동체를 구축하는 것으로 설명했다.

- “역할과 책임을 수행하는 객체” → 추상화
- “협력하는 공동체”를 구성하고 객체들로 나누는 과정 → 분해

프로그래밍 언어의 관점에서 객체지향이란 데이터를 중심으로 데이터를 추상화와 프로시저 추상화를 통합한 객체를 이용해 시스템을 분해하는 방법. 데이터 추상화와 프로시저 추상화를 함께 포함한 클래스를 이용해 시스템을 분해하는 것이다.

그렇다면 전통적인 기능 분해 방법에 비해 효과적이라고 말하는 이유는?

효과정의 이미는?

이 해답을 찾기 위해선 전통적 기능 분해 방법에서 시작해 객체 지향 분해 방법에 이르는 좌절과 극복을 봐야 한다.

### 02 프로시저 추상화와 기능 분해

**메인 함수로서의 시스템**

기능과 데이터의 첫 전쟁에서 기능이 이겼다. 기능은 오랜 시간 시스템을 분해하기 위한 기준으로 사용되었고, 이 같은 분해 방식을 알고리즘 분해 또는 기능 분해라고 했다. 기능 분해 관점에서 추상화의 단위 → 프로시저. 시스템은 프로시저 단위로 분해된다.

프로시저는 반복적으로 실행 또는 유사하게 실행되는 작업을 하나의 장소에 모아 로직을 재사용 & 중복 방지 추상화 방법이다. 내부 구현을 몰라도 인터페이스만 알면 프로시저를 알 수 있다.

프로시저는 잠재적으로 정보 은닉의 가능성을 제시하지만 프로시저만으로 효과적 정보 은닉 체계는 힘들다.

이 관점에서 시스템은 입력 값을 계산해 출력 값을 반환하는 수학의 함수와 동일하다. 시스템은 더 작은 작업으로 분해 될 수 있는 하나의 큰 메인 함수다.

전통적 기능 분해 방법은 하향식 접근(Top-Down Approach)을 따른다. 

- 최상위 기능 정의
- 최상위 기능을 좀 더 작은 하위 기능으로 분해해 나감.
- 분해는 마지막 하위 기능이 프로그래밍 언어로 구현 가능할 때까지 계속된다.
- 각 세분화 단계는 바로 위 단계보다 구체적이여야한다.

**급여 관리 시스템**

새로운 예제이다.

회사는 매달 지급해야 하는 기본급에 대해 직원과 협의해 12개월 동안 동일하게 직원들에게 지급한다. 소득 세율에 따라 일정 금액의 세금을 공제한다. 실급여는 다음 공식을 따른다.

`급여 = 기본급 - (기본급 * 세율)`

전통적이니 기능 분해 방법인 하향식 접근법을 따른다.

메인 프로시저인 최상위 문장을 적자.

`직원의 급여를 계산한다.`

구체화를 해보자. 

```
직원의 급여를 계산한다.
	사용자로부터 소득세율을 입력받는다.
	직원의 급여를 계산한다.
	양식에 맞게 결과를 출력한다.
```

더 정제 가능한 문장이 있다면 구현이 가능 할 정도의 충분히 저수준의 문장이 될 때까지 기능을 분해하자.

기본급 데이터는 시스템 내부에 보관하고, 계산 결과는 `이름: {직원명}, 급여: {계산된 금액}`으로 출력한다.

```
직원의 급여를 계산한다.
	사용자로부터 소득세율을 입력받는다.
		"세율을 입력하세요: "문장 출력
		키보드를 통해 세율 입력받음
	직원의 급여를 계산한다.
		전역 변수에 저장된 직원의 기본급 정보를 얻음
		급여를 계산
	양식에 맞게 결과를 출력한다.
		"이름: {직원명}, 급여: {계산된 금액}" 형식에 따라 출력 문자열 생성
```

급여 관리 시스템을 입력을 받아 출력을 생성하는 커다란 하나의 메인 함수로 간주하고 기능 분해를 시작했다는 점에 주목하자. 입력 정보는 직원 정보와 소득 세율이고, 출력은 계산된 급여 정보다.

기능 분해 방법은 기능을 중심으로 필요한 데이터를 결정. 메인은 기능이고, 데이터는 조연의 역할에 불과하다. 기능이 우선이고, 데이터는 기능의 뒤를 따른다.

이것은 유지보수에 다양한 문제를 야기한다. 실제로 구현해 보면서 전통적 하향식 접근의 문제점을 보자.

직원의 목록은 `employs`에, 기본급은 `basePays`에 저장한다.

```kotlin
val employees: List<String> = listOf("A", "B", "C")
val basepays: List<Int> = listOf(400, 300, 250)
```

```kotlin
fun calculatePayFor(name: String, taxRate: Float): Int {
    val index = employees.indexOf(name)
    val basePay = basePays[index]
    return (basePay - (basePay * taxRate)).toInt()
}
```

```kotlin
fun describeResult(name: String, pay: Int): String {
    return "이름: " + name + ", 급여: " + pay
}
```

**하향식 기능 분해의 문제점**

겉으로는 이상적이지만 실제로는 다양한 문제에 직면함

1. 시스템은 하나의 메인 함수로 구성되어있지 않다.
2. 기능 추가나 요구사항 변경으로 인해 메인 함수를 빈번하게 수정해야한다.
3. 비즈니스 로직이 사용자 인터페이스와 강하게 결합된다.
4. 하향식 분해는 너무 이른 시기에 함수들의 실행 순서를 고정해 유연성과 재사용성이 저하됨
5. 데이터 형식이 변경되면 파급효과를 예측할 수 없다.

설계는 코드 배치 방법이며, 설계가 필요한 이유는 변경에 대비하기 위한 것이다. 변경은 성공적인 SW가 맞이해야 하는 피할 수 없는 운명이다. 저 목록을 통해서, 하향식 접근과 기능 분해는 변경에 취약하다.

문제점을 하나씩 보자.

- 하나의 메인 함수라는 비현실적 아이디어

시스템들은 변화한다. 사용자들을 만족시키기 위해서 새로운 요구사항을 도출하고 추가한다. 이것은 하나의 메인 함수만으로 구현되는 것과 모순된다.

어느 시점에는 *유일한* 메인 함수 개념은 없어지게 될 것이다. 대부분의 시스템에서는 하나의 메인 기능이란 개념은 존재하지 않는다. 모든 기능은 규모의 측면에서는 다를 수 있지만 기능성의 측면에서는 동등하게 독립적이고 완결된 하나의 기능을 표현한다.

> 실제 시스템에 정상(top)이란 존재하지 않는다. - 버트란드 마이어
> 

- 메인 함수의 빈번한 재설계

시스템 안에 여러 개의 정상이 존재해 하나의 메인 함수를 유일한 정상으로 간주하는 하향식 기능 분해는 새로운 기능이 추가되면 메인 함수를 수정해야한다.

급여 관리 시스템에 회사에 속한 모든 직원의 기본급의 총합을 구하는 기능을 추가해 달라는 요구 사항이 들어오면 구현 자체는 어렵지 않다. 그러나 main 함수는 직원 각각의 급여를 계산하는 것이 목적이므로 전체 직원의 기본급 총액 `sumOfPays` 함수가 들어설 자리가 쉽지 않다.

이를 해결하려면 현재 메인 함수에 있는 걸 `calculatePay`로 옮기고 `sumOfBasePays`랑 `calculatePay`를 main에서 호출하는 것이다.

```kotlin
fun Main(operation: String, args: String = "") {
	when(operation) {
		is "pay" ->
			calculatePay(name)
		is "basePays" ->
			sumOfBasePays()
		else ->
			throwIllegalArgumentException()
	}
}
```

이는 정상인 메인 함수에서 출발하는 하향식 접근의 기본 가정에 문제가 있는 것을 보여준다. `sumOfBasePays`같은 함수를 추가할 때마다 `main`을 수정할 수 밖에 없다.

- 비즈니스 로직과 사용자 인터페이스의 결합

하향식 접근법은 비즈니스 로직 설계하는 초기 단계부터 입력, 출력 양식을 함께 고민하도록 강요한다. 코드 안에서 비즈니스 로직과 사용자 인터페이스 로직이 밀접하게 결합된다.

사용자 인터페이스는 시스템 내에서 자주 변경되는 부분이고, 비즈니스 로직은 사용자 인터페이스에 비해 적게 발생한다. UI가 바뀌면 비즈니스 로직도 영향을 받아 변경에 불안정한 아키텍쳐를 낳는다.

- 성급하게 결정된 실행 순서

하향식은 설계를 시작하는 시점부터 무엇을 해야하는지가 아닌, 어떻게 동작하는지에 집중하도록 한다. 직원의 급여를 계산하려면 어떤 작업이 필요한가? 소득세율을 입력받는 작업과 급여를 계산하는 작업, 결과 출력이 필요하다. 하향식 접근법의 첫 질문은 무엇이 아니라, 어떻게 이다.

메인 함수가 작은 함수들로 분해되기 위해서는 함수들의 순서를 결정해야 한다. 실행 순서를 미리 정해야 하는 것이다. 문제는 이 제어 구조가 빈번한 변경의 대상이라는 것이다. 기능 추가나 변경될 때마다 제어 구조가 변경된다. 결과적으론 기능을 추가하거나 변경 작업은 매번 기존의 함수 제어 구조를 변경하도록 한다.

하향식 접근을 통해 분해한 함수들은 상위 함수가 강요하는 문맥 안에서만 의미를 가지기 때문에 재사용하기도 어렵다. 함수가 재사용 가능하려면 상위 함수 보다 더 일반적이여야 하지만 하위 함수는 항상 상위 함수보다 문맥에 더 종속적이여서 재사용성과 반대되게 된다.

하향식 설계와 관련된 모든 문제의 원인은 결합도이다. 현재의 문맥에 강하게 결합된 시스템은 현재 문맥을 떠나 다른 문맥으로 옮기면 재사용이 어렵다. 가장 큰 문제는 전체 시스템의 핵심적인 구조를 결정하는 함수들이 데이터와 강하게 결합된다는 것이다.

- 데이터 변경으로 인한 파급효과

하향식 기능 분해의 가장 큰 단점은 어느 데이터를 어떤 함수가 사용하는지 추적이 어렵다는 것이다. 어떤 데이터가 어떤 함수에 의존하고 있는지를 파악하려면 모든 함수를 열어봐야 한다…

텍스트를 검색하는 단순한 문제가 아니다. 결합도와 의존성의 문제이다. 데이터의 변경으로 인한 영향은 데이터를 직접 참조하는 모든 함수로 퍼져 나간다.

얼마나 취약한지 이해하기 위해 새로운 기능을 추가하자. 직원 뿐만 아니라 아르바이트 또한 뽑아서 계산한다. 아르바이트는 시급에 일한 시간을 곱해야한다. 

아르바이트 직원의 이름과 시급은 `employees`와 `basePays`에 저장하고, 아르바이트 직원의 여부를 저장하는 `hourlys`를 추가한다. `true`라면 아르바이트 직원의 정보, `false`면 정규직 직원이다.

```kotlin
val employees: List<String> = listOf("A", "B", "C", "aD", "aE", "aF")
val basePays: List<Float> = listOf(400, 300, 250, 1, 1, 1.5)
val hourlys: List<Int> = listOf(false, false, false, true, true, true)
```

(성우의 코멘트 : 1.5 때문에 타입도 Float으로 바뀌었다.)

아르바이트 직원의 급여를 계산하기 위해 한달 간의 누적 업무시간을 저장해야하므로 `timeCards`에 보관한다. 정규직원은 0이다.

```kotlin
val timeCards: List<Int> = listOf(0, 0, 0, 120, 120, 120)
```

위 과정이 예제 케이스가 적지만 어마어마한 수의 함수로 구성된 거대한 시스템이라면, 전역 변수에 의존하는 함수를 찾는 것은 엄청난 시간을 요하게 될 것이다.

어떻게 해결해야할까? `calculatePay` 함수에 조건 분기를 추가해 해결 할 수 있다.

직원의 급여를 계산하는 `calculateHourlyPayFor`은 시급에 한달 일한 시간을 곱해 급여를 계산. 정규 직원과 아르바이트를 판단하는 `hourly?`함수도 추가하자. 직원이 아르바이트 직원이면 `true`를 반환한다.

```kotlin
fun calculateHourlyPayFor(name: String, taxRate: Float) {
	val index = employees.indexOf(name)
	val basePay = basePays[index] * timeCards[index]
	return basePay - (basePay * taxRate)
}

fun hourly(name:String): Boolean {
	return hourlys[employees.indexOf(name)]
}
```

`calculatePay`수정, 아르바이트 직원이면 `calculateHourlyPayFor`을 호출하고, 정직원이면 `calculatePayFor`함수를 실행한다.

```kotlin
fun calculatePay(name: String) {
	val taxRate = getTaxRate()
	val pay = 0
	if(hourly(name)) {
		pay = calculateHourlyPayFor(name, taxRate)
	} else {
		pay = calculatePayFor(name, taxRate)
	}
	
	describeResult(name, pay)
}
```

그러나… `sumOfBasePays` 값이 이상하다. `basePays`와 `employees`에 아르바이트 직원에 대한 정보를 추가했기 때문이다. `basePays`에는 정규직과 아르바이트 직원의 시급도 포함되어있기 때문에 시급을 총합에 제외해야한다.

```kotlin
fun sumOfBasePays(): Int {
	val result = 0
	for name in employees {
		if(!hourly(name)) {
			result += basePays[employees.indexOf(name)]
		}
	}
	println(result)
}
```

여기서 알 수 있는 것은 데이터 변경으로 인해 발생하는 함수에 대한 영향도를 파악하는 것이 생각보다 쉽지 않다는 것이다. 새로운 요구사항을 넣기 위해서 전역 변수를 건드리고, `calculatePay`를 수정했지만, `sumOfBasePays`도 영향을 받게 되었다.

데이터 변경으로 인한 영향을 최소화 하려면 데이터와 함께 변경되는 부분과 그렇지 않은 부분을 명확하게 분리해야한다. 이를 위해 잘 정의된 퍼블릭 인터페이스를 통해 데이터에 대한 접근을 통제해야한다.

이게 바로 의존성 관리의 핵심이다. 변경에 대한 영향을 최소화하기 위해 영향 받는 부분, 받지 안흔ㄴ 부분을 명확하게 하고 잘 정의된 퍼블릭 인터페이스를 통해 변경되는 부분에 대한 접근을 통제하자.

기능 분해가 가진 본질적 문제를 해결하기 위해 이 같은 개념을 기반으로 한 정보 은닉과 모듈이라는 개념을 제시했다.

**언제 하향식 분해가 유용한가?**

하향식 분해가 유용할 때도 있다. 설계가 어느 정도 안정화 된 후에는 설계의 다양한 측면을 논리적으로 설명하고 문서화 하기에 용이하다. 

하향식은 이미 완전히 이해된 사실을 서술하기에 적합하다. 새로운 것을 개발하고, 설계하고, 발견하는 데는 적합한 방법이 아니다. 시스템이나 프로그램 개발자가 이미 완료한 결과에 대한 명확한 아이디어를 가지고 있다면 머릿 속에 있는 것을 종이에 서술하기 위해 하향식을 사용할 수 있다. 하향식이 시작되면 문제는 이미 해결됐고, 해결돼야만 하는 세부사항이 존재할 뿐이다. - Michael Jackson <System Development>

하향식 분해는 작은 프로그램, 개별 알고리즘을 위해선 유용하다. 이미 해결된 알고리즘을 문서화하고 서술하는데는 매우 좋다.

정리

하향식 설계로 설계된 SW

- 하나의 함수에 제어가 집중되어 확장이 어렵다.
- 프로젝트 초기에 설계의 본질적인 측면을 무시하고 UI 같은 비본질적 측면에 집중하게 만든다.
- 과도하게 함수에 집중해 SW의 중요한 다른 측면인 데이터에 대한 영향도를 파악하기 어렵게 한다.
- 근본적으로 재사용이 어렵다.

### 03 모듈

**정보 은닉과 모듈**

시스템의 변경을 관리하는 기본적인 전략은 함께 변경되는 부분을 하나의 구현 단위로 묶고 퍼블릭 인터페이스를 통해서만 접근하도록 하는 것이다.

즉, 기능을 기반으로 시스템을 분해하는 것이 아니라 변경의 방향에 맞춰 시스템을 분해한다.

정보 은닉(information hinding) - 데이비드 파나스 

- 시스템에서 자주 변경되는 부분을 상대적으로 덜 변경되는 안정적인 인터페이스 뒤로 감춰야 한다는 것이 시스템을 모듈 단위로 나누기 위한 핵심이다.
- 시스템을 모듈로 분할하는 원칙은 외부에 유출돼서는 안 되는 비밀의 윤곽을 따라야 한다.
- 변경될 가능성이 있는 비밀을 내부로 감추고, 잘 정의되고 쉽게 변경되지 않을 퍼블릭 인터페이스를 외부에 제공해 내부의 비밀로 접근하지 못하게 해야한다.

시스템을 모듈로 분해한 후에는 각 모듈 내부를 구현하기 위해 기능 분해를 적용할 수 있다. 

- 기능 분해 → 기능을 구현하기 위해 필요한 기능들을 순차적으로 찾아가는 탐색 과정
- 모듈 분해 → 감춰야 하는 비밀을 선택하고 비밀 주변에 안정적인 보호막을 설치하는 보존 과정

비밀 결정 및 모듈 분해 후에는 기능 분해를 이용해 모듈에 필요한 퍼블릭 인터페이스를 구현할 수 있다.

시스템을 모듈 단위로 어떻게 분해할 것인가? 감춰야 하는 아래 두 가지 비밀을 찾아 외부에서 비밀에 접근하지 못하도록 막아라. 이 방어막이 퍼블릭 인터페이스가 된다.

1. 복잡성 - 모듈이 너무 복잡하다면 이해하고 사용하기 어렵다. 외부에 모듈을 추상화할 수 있는 간단한 인터페이스를 제공해 모듈의 복잡도를 낮추자.
2. 변경가능성 - 변경 가능한 설계 결정이 외부에 노출되면 실제로 변경이 발생했을 때 파급효과가 커진다. 변경 발생 시 하나의 모듈만 수정하면 되도록 변경 가능한 설계 결정을 모듈 내부로 감추고 외부에는 쉽게 변경되지 않을 인터페이스를 제공한다.

급여 관리 시스템 일반적 비밀 → 데이터. 변경 시 시스템을 굴복시키는 대부분의 경우는 데이터가 변경되는 경우다. (보통이라는거지, 복잡한 로직이나 변경 가능성이 큰 자료 구조 일수도 있다.) 이를 구원하는 방법도 데이터를 자신의 비밀로 삼는 모듈을 만드는 것이다. 모듈로 직원 정보는 비밀로, 외부는 퍼블릭 인터페이스만 노출.

책에는 Ruby의 `module` 키워드 사용했다. 그러나 C언어는 파일 단위로 모듈을 분리하고, 자바는 `package`, C++, C#에서는 `namespace`를 사용한다. 한 모듈로 캡슐화 해주자.(는 귀찮아서 못하겟다.ㅎ)

대략적으로 상상해보자면, 

전역 변수였던 `employees`, `basePays`, `hourlys`, `timeCards`가 `Employees`라는 모듈 내부로 숨겨진다. 모듈 외부에선 직원 정보를 관리하는 데이터에 직접 접근할 수 없다. 외부에서는 `Employees`모듈이 제공하는 퍼블릭 인터페이스에 포함된 `calculatePay`, `hourly`, `calculateHourlyPayFor`, `calculatePayFor`, `sumOfBasePays` 함수를 통해서만 내부 변수를 조작할 수 있다.

`main`함수가 `Employees`모듈의 기능을 사용하도록 코드를 수정하면 된다.

**모듈의 장점과 한계**

모듈의 장점

1. 모듈 내부 변수가 변경되도 모듈 내부에만 영향을 끼친다.
    1. 모듈 내부에 정의된 변수를 직접 참조하는 코드 위치를 모듈 내부로 제한 가능.
    2. 데이터 변경으로 인한 파급효과를 제어할 수 있어 코드를 수정하고 디버깅하기 쉽다.
2. 비즈니스 로직과 사용자 인터페이스 관심사를 분리한다.
    1. 사용자 입력과 출력을 `Employees`가 아닌, 외부에 있다. `Employees`는 비즈니스 로직만 담당하고, UI 관심사는 모두 `main`에 위치한다.
3. 전역변수와 전역함수 제거로 네임스페이스 오염(namespace pollution)을 방지한다.
    1. 모듈의 한 가지 용도는 네임스페이스를 제공하는 것이다. 변수와 함수를 모듈 내부에 포함시켜 다른 모둘에서도 동일한 이름을 사용할 수 있게 된다. 네임스페이스의 오염을 방지하고, 이름 충돌의 위험을 완화한다.

모듈은 기능이 아니라 변경의 정도에 따라 시스템을 본해하게 한다. 각 모듈은 외부에 감춰야할 비밀과 관령성 높은 데이터와 함수의 집합이다. 따라서 모듈 내부는 높은 응집도를 유지한다. 모듈과 모듈 간에는 퍼블릭 인터페이스를 통해서만 통신해야한다. 따라서 낮은 결합도를 유지한다.

모듈의 핵심은 데이터다. 감춰야 할 데이터가 뭔지, 필요한 함수는 뭔지 생각한다. 기능이 아니라, 데이터를 중심으로 시스템을 분해하는 것이다.

모듈이 프로시저 추상화 보다는 높은 추상화 개념을 제공하지만, 태생적으로 변경을 관리하기 위한 구현기법이라 추상화 관점의 한계점이 명확하다. 좀 더 높은 수준의 추상화를 위해선 직원 전체가 아니라, 개별 직원을 독립적 단위로 다룰수 있어야 한다.

이를 만족시키기 위해 등장한 개념이 추상 데이터 타입이다.

### 04 데이터 추상화와 추상 데이터 타입

**추상 데이터 타입**

프로그래밍 언어에서 타입은… 변수에 저장할 수 있는 내용물의 종류와 변수에 적용될 수 있는 연산의 가짓수다. 언어는 다양한 형태의 내장 타입(built-in type)을 제공한다. 기능 분해 시절에 사용한 절차형 언어는 새로운 타입을 추가하는게 어려웠다. 주된 추상화는 프로시저 추상화였다. 시간이 지나서 사람들은 프로그램의 표현력을 향상시키는데 한계가 있음을 발견했다.

데이터 추상화(Data abstraction) - 바바라 리스코프(Barbara Liskov)

프로시저 만으론 충분히 풍부한 추상화를 제공할 수 없다. 이는 언어 설계에서 가장 중요한 추상 데이터 타입(Abstract Data Type)의 개념으로 인도했다. 이는 추상 객체의 클래스를 정의한 것으로 추상 객체에 사용할 수 있는 오퍼레이션을 이용해 규정된다. 추상 데이터 객체를 사용할 때 프로그래머는 오직 객체가 외부에 제공하는 행위에만 관심을 가지며 행위가 구현되는 세부적 사항에 대해선 무시한다. 객체가 저장소 내에서 어떻게 표현되는지와 같은 구현 정보는 오직 오퍼레이션을 어떻게 구현할 것인지에 집중할 때만 필요하다. 객체의 사용자는 이 정보를 알거나 제공받을 필요가 없다.

- 데이터 추상화, 정보 은닉, 데이터 캡슐화, 인터페이스-구현분리 개념이 모두 있다.
- 추상 데이터 타입은 프로시저 추상화 대신 데이터 추상화를 기반으로 SW를 개발하게 한 첫걸음이다. 이를 지원하려면 아래와 같은 프로그래밍 언어의 지원이 있어야 한다.
    - 타입 정의를 선언할 수 있어야 한다.
    - 타입의 인스턴스를 다루기 위해 사용할 수 있는 오퍼레이션의 집합을 정의할 수 있어야 한다.
    - 제공된 오퍼레이션을 통해서만 조작할 수 있도록 데이터를 외부로부터 보호할 수 있어야 한다.
    - 타입에 대해 여러 개의 인스턴스를 생성 가능해야한다.

추상 데이터 타입을 구현할 수 있는 언어적 장치를 제공하지 않는 언어에서도 추상 데이터 타입을 구현하는 것은 가능은 하다. 다만 좀 그럴뿐…

추상 데이터 타입을 이용해 급여 관리 시스템을 고쳐보자. 루비에는 추상 데이터 타입을 따라할 수 있는 `Struct`키워드가 있다. 루비는 몰라서 일단 대략적으로 따라가보자

```ruby
Employee = Struct.new(:name, :basePay, :hourly, :timeCard) do
end
```

`name`, `basePay`,  `hourly`, `timeCard`를 가지는 추상 데이터 타입을 만든다. 이는 비밀로 만들어진다.

추상 데이터 타입에 적용할 오퍼레이션은 `calculatePay`, `calculateHourlyPay`, `calculateSalariedPay`를 선언해준다. 기본급 계산 함수도 추가하자(`monthlyBasePay`).

추상 데이터 타입은 사람들이 세상을 바라보는 방식에 좀 더 근접하게 추상화 수준을 향상시킨다. 이는 말 그대로  상태를 저장할 데이터를 표현한다. 데이터에 대한 관점을 설계의 표면으로 끌어올리지만, `main`에 묶여있는 건 여전하다. 데이터와 기능을 분리하는 절차인 설계의 틀에 갇혀있다.

그럼… 클래스는 추상 데이터 타입인가?

### 05 클래스

**클래스는 추상 데이터 타입인가?**

명확한 의미에서는 추상 데이터 타입과 클래스는 동일하지 않다. 핵심적 차이는 클래스는 상속, 다형성을 지원하지만 추상 데이터 타입은 지원하지 못한다. 상속과 다향성을 지원하는 OOP와 구분하기 위해 상속과 다형성을 지원하지 않는 추상 데이터 타입 기반의 프로그래밍 패러다임을 객체기반 프로그래밍(Object-Based Programming)이라고도 한다. 

윌리엄 쿡은 추상 데이터 타입은 타입을 추상화 한 것, 클래스는 절차를 추상화 한 것이라고 설명함.

두 개를 이해 하기 위해 `Employee` 타입의 `calculatePay`와 `monthlyBasePay` 오퍼레이션을 보자. `Employee` 는 물리적으론 하나지만, 개념적으론 정규 직원, 아르바이트를 포괄하는 복합 개념이다. `calculatePay`와 `monthlyBasePay`는 각각 동작한다. `Employee`가 정규직을 나타내면 `calculatePay`가 나오지만, 알바면 그냥 곱셈 세액만 계산한다.

!image.png

하나의 타입처럼 보이는 `Employee`내부에는 정규 직원과 알바가 공존하는 것이다. 설계 관점에서 직원 타입을 외부에 캡슐화 하는 것이다. 하나의 대표 타입이 구체적 타입을 감추기 때문에 이를 타입 추상화라고 불렀다. 개별 오퍼레이션이 모든 개념적인 타입에 대한 구현을 포함하도록 해 하나의 물리적 타입 안에 전체를 가둔다.

!image.png

타입 추상화를 기반으로 하는 대표적 기법이 추상 데이터 타입이다. 위 그림은 추상 데이터 타입인 `Employee`가 어떻게 오퍼레이션을 기준으로 각각을 묶는지 표현한다. `calculatePay`와 `monthlyBasePay` 를 호출 가능하지만, 정규 직원이나 아르바이트 직원이 있는 건 알 수 없다.

객체 지향은 타입을 기준으로 오퍼레이션을 묶는다. 정규 직원과 아르바이트 두 개의 타입을 명시적으로 정의하고, 두 직원 유형과 관련된 오퍼레이션의 실행 절차를 두 타입에 분배한다. 따라서, 각 클래스들이 `calculatePay` 와 `monthlyBasePay`를 적절하게 구현하게 된다. 이는 상속을 통해 공통 로직을 포함하는 부모 클래스를 정의 가능하다.

클라이언트의 관점에서 두 클래스의 인스턴스는 동일하나, 실제 내부 절차는 다르지만 절차에 대한 차이점을 준다. 객체 지향은 절차 추상화(procedural abstraction)이다.

!image.png

추상 데이터 타입 → 오퍼레이션 기준 타입 추상화

클래스 → 타입 기준 절차 추상화

**추상화 데이터 타입에서 클래스로 변경하기**

`Employee`는 추상 데이터 타입으로는 하나로 나오지만, 클래스는 두 개가 나온다. 추상 데이터 타입에선 `Employee`는 두 타입을 모두 구현하지만, 클래스에서는 각 직원이 공통적으로 가져야 할 메서드 시그니처만 정의한다.

```java
abstract class Employee {
	private String name;
	private float basePay;
		
	public Employee(String name, float basePay) {
		this.name = name;
		this.basePay = basePay;
	}
	
	public abstract float calculatePay(float taxRate) ;
	public abstract float monthlyBasePay() ;
}
```

정규 직원을 `SalariedEmployee`, 아르바이트를 `HourlyEmployee` 로 추가한다.

```java
class SalariedEmployee extends Employee {
	public SalariedEmployee(String name, float basePay) {
		this.name = name;
		this.basePay = basePay;
	}
	
	public float calculatePay(float taxRate) {
		return basePay - (basePay * taxRate);
	}
	
	public float monthlyBasePay() {
		return basePay;
	}
}

// HourlyEmployee도 같이 구현.
```

객체를 생성하고 나면 클래스가 무엇인지는 중요하지 않다. `Hourly~`, `Sal~` 둘다 `Employee`처럼 다룰 수 있다. 

**변경을 기준으로 선택하라**

단순히 클래스를 구현 단위로 사용하는 것이 OOP를 한다는 것을 의미하지 않는다. 타입을 기준으로 절차를 추상화 하지 않으면 그건 OO 분해가 아니다. 클래스를 사용하고 있더라도 말이다. 

클래스가 추상 데이터 타입을 따르는지 확인하는 것은 클래스 내부에 인스턴스의 타입을 표현하는 변수가 있는지 확인하는 것이다. 

객체 지향에서는 타입 변수를 이용한 조건문을 다형성으로 대체한다. 클라이언트가 객체의 타입을 확인한 후 적절한 메서드 호출이 아닌 객체가 메시지를 처리할 메서드를 선택한다.

모든 설계 문제가 그러하듯, 조건문을 기피하는 이유는 변경 때문이다. 추상 데이터 기반으로 한 `Employee` 에 새로운 타입을 추가하기 위해선 `hourly`의 값을 체크하는 클라이언트의 조건문을 하나씩 찾아 다 수정해야한다.

이에 반해 객체 지향은 새로운 직원 유형을 구현하는 클래스를 `Employee`상속 계층에 추가하고 메서드를 오버라이딩 하면 된다. 즉, 시스템에 새로운 로직을 추가하기 위해 클라이언트 코드를 수정할 필요가 없다.

이처럼 기존 코드에 아무 영향 없이 새로운 객체 유형과 행위를 추가할 수 있는 객체 지향의 특성을 개방-폐쇄 원칙(Open-Closed Principle, OCP)라고 한다. 이것은 객체지향 설계가 전통적 방식에 비해 변경하고 확장하기 쉬운 구조를 설계할 수 있는 이유다.

그럼 추상 데이터 타입은 모든 경우에 최악의 선택인가? 

설계의 유용성은 변경의 방향성과 발생 빈도에 따라 결정된다. 추상 데이터 타입과 객체 지향 설계의 유용성은 설계에 요구되는 변경의 압력이 ‘타입 추가’에 관한 것인지, ‘오퍼레이션 추가’에 관한 것인지에 따라서 달라진다.

- 타입 추가라는 변경의 압력이 강한 경우, 객체지향의 손을 들어줘야 한다. 추상 데이터 타입의 경우 새 타입을 추가하려면 타입을 체크하는 코드들을 다 일일이 찾아 수정, 테스트 해야하지만, 객체지향은 그럴 필요가 없다.
- 오퍼레이션 추가라는 변경의 압력이 강한 경우, 추상 데이터 타입의 손을 들어줘야 한다. 객체지향의 경우, 새 오퍼레이션을 추가하려면 상속 계층에 속하는 모든 클래스를 한번에 수정해야한다. 그러나 추상 데이터 타입에서는 새로운 오퍼레이션 추가 작업이 쉽다.

새로운 타입을 빈번하게 추가한다면 객체지향, 새로운 오퍼레이션을 빈번하게 추가하면 추상 데이터 타입. 객체지향적 설계가 모든 경우에 정답은 아니다.

**협력이 중요하다**

단순히 오퍼레이션과 타입을 표에 적어놓고 클래스에 오퍼레이션 구현 방법을 분해한다고 객체 지향적 설계를 갖는게 아니다. 객체 지향의 핵심은 역할, 책임, 협력이다. 협력을 고려하지 않고 객체를 고립시킨 채 오퍼레이션의 구현 방식을 타입별로 분배하는 것은 옳지 않다.

위의 내용들은 추상 데이터 타입 vs 클래스인 거지, 클래스 설계는 3장의 책임 주도 설계를 따라야 한다.

객체가 참여할 협력을 정하고, 협력에 필요한 책임을 수행하려면 어떤 객체가 필요한지 고민하라. 책임을 다양한 방식으로 수행해야 할 때만 타입 계층 안의 절차를 추상화 하자. 타입 계층과 다형성은 협력이라는 문맥 안에서 책임을 수행하는 방법에 관해 고민한 결과물이여야 하며, 그 자체가 목적이 되면 안된다.
