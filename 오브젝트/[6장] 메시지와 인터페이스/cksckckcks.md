# 메시지와 인터페이스

- 훌륭한 객체 지향 코드를 얻으려면 → 협력 안에서 객체가 수행하는 책임에 초점을 맞춰야 함

→ 책임이 객체가 수신할 수 있는 메시지의 기반이 된다는 것

- 객체가 수신하는 메시지들이 객체의 퍼블릭 인터페이스를 구성

### 클라이언트 - 서버 모델

- 협력 안에서 메시지를 전송하는 객체를 클라이언트, 메시지를 수신하는 객체를 서버라고 함
- 협력은 클라이언트가 서버의 서비스를 요청하는 단방향

### 메시지와 메시지 전송

- 메시지는 객체들이 협력하기 위해 사용할 수 있는 유일한 의사소통 수단
- 객체가 다른 객체에게 도움 요청 : 메시지 전송 / 메시지 패싱
- 이를 전송하는 객체 : 메시지 전송자
- 이를 수신하는 객체 : 메시지 수신자
- 메시지의 구성 : 오퍼레이션 이름과 인자(argument)

### 메시지와 메서드

- 메시지의 수신자의 실제 타입이 무엇인가에 따라 어떤 코드가 실행되는지 결정
- 메서드 : 메시지를 수신햇을 때 실제로 실행되는 함수 또는 프로시저

### 퍼블릭 인터페이스와 오퍼레이션

- 객체는 안과 밖을 구분하는 뚜렷한 경계를 가진다.
- 외부의 객체 → 오직 객체가 공개하는 메시지를 통해서만 객체와 상호작용
    - 퍼블릭 인터페이스 : 객 체가 의사소통을 위해 외부에 공게하는 메시지의 집합
- 오퍼레이션 : 프로그래밍 언어의 관점에서 퍼블릭 인터페이스에 포함된 메시지

### 디미터 법칙

- 객체는 자신이 직접 알고 있는 객체와만 상호작용해야 하며, 낯선 객체의 내부 구조를 몰라야 한다
- 낯선자에게 말하지 말라 (오직 인접한 이웃이랑만 말해라)
- 오직 하나의 도트만 사용하라

→ 부끄럼타는 코드 작성 가능

### 묻지 말고 시켜라

- 메시지 전송자는 메시지 수신자의 상태를 기반으로 결정을 내린 후 메시지 수신자의 상태를 바꿔서는 안됨
- 훌륭한 인터페이스를 수확하기 위해서는 객체가 어떻게 작업하는지 노출 X

### 의도를 드러내는 인터페이스

- 메서드의 이름 명짓기
    - 메서드가 작업을 어떻게 수행하는지를 나타내도록 이름 짓기
    - 어떻게가 아니라 무엇을 하는지를 드러내도록 이름 짓기

### 디미터 법칙은 하나의 도트(.)를 강제하는 규칙이 아니다

- 하나 이상의 도트(.)를 사용하는 모든 케이스가 디미터 법칙 위반인 것은 아니다. 객체 내부 구현에 대한 어떤 정보도 외부로 노출하지 않는다면 그것은 디미터 법칙을 준수한 것이다.
