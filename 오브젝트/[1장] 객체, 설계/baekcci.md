# 01. 객체, 설계

### "이론이 먼저일까, 실무가 먼저일까?"

- 이론을 정립할 수 없는 초기에는 실무가 먼저 급속한 발전을 이룬다.
  - 실무가 어느 정도 발전 -> 실무의 실용성을 입증할 수 있는 이론이 서서히 확립된다.

=> 아무것도 없는 상태에서 이론을 정립하기 보다는 실무를 관찰한 결과를 바탕으로 이론을 정립하는 것이 최선

: **이론보다 실무가 먼저다.**

### 소프웨어 모듈의 세가지 목적

1. 실행 중 제대로 동작하는 것
2. 변경 위해 존재하는 것
3. 코드를 읽는 사람과 의사소통 하는 것

### 예상을 빗나가는 코드

- 이해 가능한 코드 == 예상에서 크게 벗어나지 않는 코드

### 변경에 취약한 코드

- 변경에 취약한 코드 feat. 객체 사이의 **의존성**
  - 객체 사이의 의존성 완전 제거X, 서로 의존하며 협력하는 객체들의 공동체 구축O
- 결합도(Coupling): 객체 사이의 의존성


: 따라서, 기능 구현에 필요한 **최소한의 의존성만 유지**하고 **불필요한 의존성을 제거**하는 것을 목표로 함

### 자율성을 높이자

- 캡슐화: 개념적/물리적으로 객체 내부의 세부적인 사항을 감추는 것
  - 목적: 변경하기 쉬운 객체를 만드는 것
  - 객체 내부로의 접근을 제한하면 객체간의 결합도를 낮출 수 있기 때문에 설계 변경에 용이함 
- 객체를 **인터페이스**와 **구현**으로 나누고 인터페이스만 공개하기
  - 객체 사이의 결합도를 낮추고 변경하기 쉬운 코드를 작성하기 위한 기본 적인 설계 원칙
 
### 캡슐화와 응집도

- 응집도: 밀접하게 연관된 작업만 수행, 연관성 없는 작업은 다른 객체에게 위임하는 객체
  - 자신의 데이터를 스스로 처리하는 자율적인 객체를 만들면 결합도를 낮추고 응집도를 높일 수 있다.

### 절차지향과 객체 지향

- 절차지향 프로그래밍: 프로세스와 데이터를 별도의 모듈에 위치시키는 방식
  - 프로세스가 필요한 모든 데이터에 의존해야 한다는 근본적인 문제점이 있음 -> 변경에 취약
- 객체지향 프로그래밍: 데이터와 프로세스를 동일한 모듈 내부에 위치시키는 방식
  - 캡슐화를 이용해 의존성을 적절히 관리 -> 결합도를 낮춤
  - 자신의 문제를 스스로 처리하기 때문에 이해하기 쉬움
  - 객체 내부의 변경이 객체 외부에 파급되지 않도록 제어할 수 있기 때문에 변경이 수월
 
### 책임(기능)의 이동

- 각 객체는 자신을 스스로 책임진다.
- 설계를 어렵게 만드는 것은 의존성
  - 불필요한 의존성을 제거함으로써 결합도를 낮춤
  - 결합도를 낮추기 위해 캡슐화
  - 캡슐화하는 것은 객체의 자율성을 높이고 응집도 높은 객체들을 만들 수 있게 함.
 
### 설계가 왜 필요한가

- 설계란 코드를 배치하는 것
- 좋은 설계란?: 변경을 수용할 수 있는 설계
  - 요구사항이 항상 변경하기 때문
  - 코드 변경 시 버그가 추가될 가능성이 높기 때문
