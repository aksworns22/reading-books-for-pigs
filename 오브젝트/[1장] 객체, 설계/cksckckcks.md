# 객체, 설계

## 이론이 먼저일까? 실무가 먼저일까?

- 대부분 사람들은 이론이 먼저 정립된 후 실무가 발전한다고 생각.
- 글래스는 실무가 먼저 급속한 발전을 이루고 나서야 이론이 서서히 모습을 갖춰가기 시작 → 성숙해지는 시점에 이론이 실무를 추월한다고 주장 (이론보다 실무가 먼저)
- 현재 소프트웨어는 이론보다 실무가 더 앞서 있다. (실무가 더 중요)

→ 소프트웨어 설계, 소프트웨어 유지보수

## 티켓 판매 애플리케이션 구현하기

### 소극장 관련 코드

- 관람객은 초대권을 받은 관람객, 받지 않은 관람객이 있다.
    - 초대권을 받은 관람객 : 초대권 → 티켓 제공
    - 초대권을 받지 못한 관람객 : 현금으로 티켓 구입 → 티켓 제공

## 무엇이 문제인가

### 소프트웨어 모듈 3가지 목적

1. 실행 중 제대로 동작하는 것 (모듈의 존재 이유)
2. 변경을 위해 존재하는 것 (간단한 작업만으로도 변경 가능 == 변경에 용이)
3. 코드를 읽는 사람과 의사소통 하는 것 (특별한 훈련 없이도 개발자가 쉽게 읽고 이해 가능해야 함)

→ 소극장 코드는 1번은 만족하지만 2, 3번은 만족하지 못함

### 예상을 빗나가는 코드

- 관람객과 판매원이 소극장의 통제를 받는 수동적인 존재라는 문제
    - 소극장은 관람객의 가방을 열어 초대장이 있는지 확인 → 왜 가방을 열어봐? (왜 가방을 너가 접근할 수 있어?)
    - 초대장이 있으면 판매원은 매표소에 보관돼 있는 티켓을 관람객의 가방으로 옮김 → 왜 티켓을 너가 넣어줘?
    - …
- 세부적인 내용을 모두 기억하고 있어야 함
    - 관람객이 가방을 가지고 있고, 가방 안에는 티켓과 현금이 잇고, 티켓 셀러가 티켓 오피스에서 티켓을 판매하고… → 머리터진다 복잡하다
- 하나를 변경할 경우 다른 것들도 모두 변경해야되는 상황

### 변경에 취약한 코드

위 코드는 관람객이 현금과 초대장을 보관하기 위해 가방을 들고 다닌다고 가정, 판매원이 매표소에서만 티켓을 판매한다고 가정

- 관람객이 가방을 들고 있지 않으면? 현금이 아니라 신용카드라면? 키오스크가 생긴다면?
    
    → 모든 코드가 흔들린다. == **의존성**과 관련된 문제다!
    
- 의존성은 변경에 대한 영향을 암시한다. (이게 변경되면 의존한 것도 변경될 수 있다)
    
    → 그렇다고 모든 객체 의존성을 제거하는 것이 정답은 아니다. 최소한의 의존성만 유지하고 불필요한 의존성을 제거하자
    
- 객체 사이의 의존성이 과한 경우를 가리켜 **결합도**가 높다고 말한다.
    
    → 객체 사이의 결합도를 낮춰 변경이 용이한 설계를 만들자
    

## 설계 개선하기

위 코드는 이해하기 힘들다. Theater가 관람객의 가방과 판매원의 매표소에 직접 접근하기 때문. == 관람객과 판매원이 자신의 일을 스스로 처리해야 한다는 우리의 직관을 벗어남 (의사소통 불가)

- Theater가 Audience와 TicketSeller에 관해 너무 세세한 부분까지 알지 못하도록 정보를 차단하면 해결된다.
- 관람객이 가방을 가지고 있다. → 이걸 영화관이 알아야되나? X
    
    → 관람객과 판매원을 **자율적인 존재**로 만들자
    

### 자율성을 높이자

지금 코드는 Theater가 거의 모든 것을 마음대로 접근할 수 있다.

→ 각 객체가 자율성을 높히도록 **캡슐화**를 진행하고, **인터페이스**에만 의존하는 구조를 만들자

### 캡슐화와 응집도

- 핵심은 객체 내부의 상태를 캡슐화하고 객체 간에 오직 메시지를 통해서만 상호작용하도록 만드는 것
- 밀접하게 연관된 작업만을 수행하고 연관성 없는 작업은 다른 객체에게 위임하는 객체를 가리켜 응집도가 높다고 말한다. → Good
- 자신의 데이터를 스스로 처리하고, 책임지게 만들면 응집도를 높힐 수 있다.

### 절차지향과 객체지향

- 절차적 프로그래밍 : 프로세스와 데이터를 별도의 모듈에 위치시키는 방식 (Theater가 모든 데이터(Bag, TicketSelle, …)에 의존 → 우리의 직관에 위배 (스스로 처리 X) → 변경에 취약
- 객체지향 프로그래밍 : 데이터와 프로세스가 동일한 모듈 내부에 위치
    
    → 훌륭한 객체 지향 설계의 핵심은 캡슐화를 통해 의존성을 적절히 관리 + 객차 사이 결합도를 낮추는 것
    

### 책임의 이동

- 코드 수정 전 설계에서는 Theater가 전체적인 작업 처리 → 변경 후에는 객체가 맡은 일을 스스로 처리
    
    → 책임의 이동 (각 객체는 자신을 스스로 책임짐)
    
- 설계를 어렵게 만드는 것은 의존성이다. → 결합도를 낮춰서 해결한다.
- 캡슐화하는 것은 객체의 자율성을 높이고 응집도 높은 객체들의 공동체를 창조할 수 있게 한다.

### 더 개선할 수 있다

- 설계하는 방법은 한가지 이상일 수 있다.
- 동일한 기능을 한가지 이상의 방법으로 설계할 수 있기 때문에 결국 설계는 트레이드오프의 산물이다.
    
    → 어떤 경우에도 모든 사람들을 만족시킬 수 있는 설계를 만들 수는 없다.
    
    → 설계는 균형의 예술이다. 훌륭한 설계는 적절한 트레이드오프의 결과물.
    

### 그래, 거짓말이다!

우리가 세상을 바라보는 직관과 일치하는 코드는 이해하기 더 쉬운 경향이 있다.

- Theater, Bag는 혼자 일 못하잖아? (무생물) → 현실에서는 수동적인 존재라고 하더라도 일단 객체지향의 세계에 들어오면 모든것이 능동적이고 자율적으로 바뀐다. (**의인화**)

## 객체 지향 설계

- 우리가 원하는 것 : 변경에 유연하게 대응할 수 있는 코드
    
    → 객체지향 프로그래밍은 의존성을 효율적으로 통제할 수 있는 방법 제공 == 요구사항 변경에 좀 더 수월하게 대응할 수 있는 가능성을 높여줌
    
- 변경 가능한 코드 : 이해하기 쉬운 코드
