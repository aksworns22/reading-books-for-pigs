# 1장: 객체, 설계

### 이론 vs 실무

- 뭐가 먼저인가?
- 대부분의 사람들은 이론 다음에 실무가 따라온다고 생각하지만, 글래스 씨는 어떤 분야이던 이론을 정립할 수 없는 초기에는 실무가 먼저 급속한 발전을 이루고, 이론은 실무가 어느정도 발전하고 난 뒤에 급격히 발전하여 해당 분야가 성숙해지는 때에 추월한다고 봄
    
    > 일단 게임은 그런 것 같은데요 ㄷㄷ
    > 
- 그렇다면 소프트웨어 분야는 어디에 있을까?
    - 아직 이론이 실무를 넘지 못했다.
- 대표적인 예시로 설계와 유지보수가 있을  것

### 티켓 판매 프로그램 구현하기

- 처음에는 다음과 같은 코드를 구현하였다
    - 소극장이 먼저 관람객의 가방 안에 초대장이 들어있는지 확인한다.
    - 초대장이 들어있다면 이벤트에 당첨된 관람객으로 판단, 티켓을 관람객의 가방 안에 넣어준다.
    - 가방 안에 초대장이 없다면 티켓을 판매한다.
        - 소극장은 관람객의 가방에서 티켓 금액을 차감한 뒤 티켓을 넣는다.
- 해당 설계는 문제가 있다.

### 무엇이 문제인가?

- 소프트웨어 모듈이 가져야 하는 세 가지 기능
    - 실행 중에 제대로 동작하는 것
    - 변경을 위해 존재하는 것
    - 특별한 훈련 없이도 개발자가 쉽게 읽고 이해할 수 있어야 함
- 근데 앞의 프로그램은 관광객을 입장시키는 데에 필요한 기능들을 다 하고 있지 않는가?
    - 하지만 변경 용이성, 가독성은 만족하지 않음!

#### 예상을 빗나가는 코드

- 관람객과 판매원이 소극장의 통제를 받는 수동적인 존재.
    - 이는 문제이다.
- 관람객의 입장에서 소극장이 허락도 없이 관람객의 가방을 뒤지고 돈을 가져간다…
- 판매원의 입장에서도 소극장이 티켓과 현금에 마음대로 접근…
    - 일반적인 상식에 맞지 않다.
    
    > 근데 소프트웨어 세계는 현실 세계랑 다르다고 하지 않았나..?
    > 
- 또한 해당 코드를 이해하기 위해서는 여러가지 세부적인 내용을 한꺼번에 기억하고 있어야 함..
    - 하나의 클래스나 메서드에서 너무 많은 세부사항을 다루기 때문에…

#### 변경에 취약한 코드

- 더 큰 문제
    - 변경에 취약
    - 관람객의 내부를 다른 객체들이 많이 알면 알수록, 관람객 클래스를 변경하기 어려워짐…
- 그렇다면 의존성을 완전히 없애는 것이 정답인가?
    - 아니다, 최소한의 의존성은 반드시 있어야 한다.
    - 설계의 목표는 객체

### 설계 개선하기

- 변경과 의사소통
    - 코드를 이해하기 어려운 이유: 소극장이 관람객의 가방과 판매원의 매표소에 직접 접근하기 때문,
        - 관람객과 판매원이 자신의 일을 스스로 처리해야 한다는 직관을 벗어남

#### 자율성을 높이자

- 설계를 변경하기 어려운 이유는 소극장이 관람객과 직원 뿐만 아니라 관람객 소유의 가방, 티직원이 근무하는 티켓까지 마음대로 접근할 수 있기 때문이다…
    - 해결방안: 각 객체가 자율적인 존재가 되도록 설계를 변경하기

#### 무엇이 개선됐는가

- 수정된 관람객과 직원은 자신이 가지고 있는 소지품을 스스로 관리한다.
- 관객이나 직원의 내부 구현을 변경하더라도 소극장을 함께 변경할 필요가 없어짐!

#### 어떻게 한 것인가

- 판매자가 티켓을 판매하기 위해 티켓 오피스를 사용하는 모든 부분을 직원 내부로 옮기고 관람객이 티켓을 구매하기 위해 가방을 사용하는 모든 부분을 관객 내부로 옮긴 것
- 결론적으로 자기 자신의 문제를 스스로 해결하도록 코드를 변경한 것.

#### 캡슐화와 응집도

- 핵심은 객체 내부의 상태를 캡슐화하고 객체 간의 오직 메시지를 통해서 상호작용하도록 만드는 것
- 단지 자신의 메시지에 응답할 수 있고 자신이 원하는 결과를 반환할 것이라는 사실을 알고 있을 뿐

#### 절차지향과 객체지향

- 일반적으로 절차적 프로그래밍 방식으로 작성된 코드의 전형적인 의존성 구조를 보여줌
- 변경하기 쉬운 설계:
    - 한 번에 하나의 클래스만 변경할 수 있는 설계
- 훌륭한 객체지향 설계의 핵심은 캡슐화를 이용해 의존성을 적절히 관리함으로써 객체 사이의 결합도를 낮추는 것

#### 책임의 이동

- 책임은 기능이다.
- 객 객체는 자신을 스스로 책임진다
- 객체지향 애플리케이션은 스스로 책임을 수행하는 자율적인 객체들의 공동체를 구성함으로써 완성된다
- 의존성은 설계를 어렵게 만든다

#### 더 개선할 수 있다

- 구현이 아닌 인터페이스에게만 의존하게…
- 전체 설계 관점에서는 결합도 상승 vs 티켓 오피스의 자율성 상승
    - 이러한 트레이드 오프의 시점이 올 수 있다.
- 훌륭한 설계는 적절한 트레이드 오프의 결과물이라는 사실을 명심하자

#### 그래 거짓말이다!

- 우리가 세상을 바라보는 직관: 객체가 스스로 책임을 지는 것
- 하지만 실세계에서는 그런가?
    - 아니다. 무생물은 스스로 행동할 수 없다.
- 현실에서는 수동적인 존재라고 하더라도 일단 객체지향의 세계에 들어오면 모든 것이 능동적이고 자율적인 존재로 바뀐다.

#### 설계가 왜 필요한가?

- 좋은 설계란 무엇인가?
    - 오늘 요구하는 기능을 온전히 수행하면서 내일의 변경을 매끄럽게 수용할 수 있는 설계이다
    - 변경을 수용할 수 있는 설계가 중요한 또 다른 이유는 코드를 변경할 때 버그가 추가될 가능성이 높기 때문이다.

#### 객체지향 설계

- 결국 우리가 진정으로 원하는 것: 변경에 유연하게 대응할 수 있는 코드
    - 객체지향은 의존성을 효율적으로 통제할 수 있는 방법을 제공함으로써 요구사항 변경에 좀 더 수월하게 대응할 수 있는 가능성을 높여준다.
- 변경 가능한 코드는 이해하기 쉬운 코드다
- 객체지향 패러다임은 세상을 바라보는 방식대로 코드를 작성할 수 있도록 돕는다.
