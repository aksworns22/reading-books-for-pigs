# 객체지향 프로그래밍

### 협력, 객체, 클래스
1. 어떤 클래스가 필요한지를 고민하기 전에 어떤 객체들이 필요한지 고민하라.
    - 클래스는 상태와 행동을 공유하는 객체들을 추상화한 것으로 먼저 객체가 어떤 상태와 행동을 가지는지를 결정해야 함
2. 객체를 독립적인 존재가 아니라 기능을 구현하기 위해 협력하는 공동체의 일원으로 봐야 한다.
    - 공통된 특성과 상태를 가진 객체들을 타입으로 분류하고 이 타입을 기반으로 클래스를 구현

### 도메인과 클래스 구현
- 도메인: 문제를 해결하기 위해 사용자가 프로그램을 사용하는 분야
    - 일반적으로 클래스 이름은 대응되는 도메인 개념의 이름과 동일하거나 적어도 유사하게 지어야 함
    - 클래스 사이의 관계도 최대한 도메인 개념 사이에 맺어진 관계와 유사하게 만들어 프로그램 구조를 이해하고 예상하기 쉽게 만들어야함
- 클래스는 내부와 외부로 구분되면 훌륭한 클래스를 설계하기 위해서는 어떤 것을 외부에 공개하고 어떤 것을 감출지를 잘 결정해야 함
    - 경계의 명확성이 객체의 자율성을 보장하기 때문에 내부와 외부로 구분해야함

### 자율적인 객체
1. 객체가 상태와 행동을 함께 가지는 복합적인 존재라는 것
2. 객체가 스스로 판단하고 행동하는 자율적인 존재하는 것

- 캡슐화: 데이터와 기능을 객체 내부로 함께 묶는 것
- 접근제어: 외부에서의 접근을 통제할 수 있는 것

--> 캡슐화와 접근 제어는 객체를 두 부분으로 나눔(`Public Interface`와 `Implementation`)
1. Public Interface: 외부에서 접근 가능한 부분 
2. Implementation: 내부에서만 접근 가능한 부분

- 객체는 보통 상태는 숨기고 행동만 외부에 공개해야 함

### 협력
- 시스템의 어떤 기능을 구현하기 위해 객체들 사이에 이뤄지는 상호작용
    - 객체가 다른 객체와 상호작용할 수 있는 유일한 방법은 메시지를 전송하는 방법뿐임

### 컴파일 시간 의존성과 실행 시간 의존성
- 코드의 의존성과 실행 시점의 의존성이 다르면 코드를 이해하기 어려워짐
- 코드의 의존성과 실행 시점의 의존성이 다를수록 코드는 더 유연해지고 확장 가능해진다

--> 의존성의 양면성

### 다형성
- 동일한 메시지를 수신했을 때 객체의 타입에 따라 다르게 응답할 수 있는 능력
    - 메시지에 응답하기 위해 실행될 메서드를 컴파일 시점이 아닌 실행 시점에 결정한다. (`지연 바인딩` or `동적 바인딩` )

### 인터페이스
- 객체가 이해할 수 있는 메시지의 목록의 정의하는 것
    - 상속을 통해 자식 클래스는 자신의 인터페이스에 부모 클래스의 인터페이스를 포함하게 됨

### 추상화의 장점
1. 추상화의 계층만 따로 떼어놓고 살펴보면 요구사항의 정책을 높은 수준에서 서술할 수 있음
    - 세부적인 내용을 무시한 채 상위 정책을 쉽고 간단하게 표현할 수 있음
2. 추상화를 이용하면 설계가 좀 더 유연해짐
    - 설계가 구체적인 상황에 결합되는 것을 방지함

### 합성
- 인터페이스에 정의된 메시지를 통해서만 코드를 재사용하는 방법
    - 구현을 효과적으로 캡슐화할 수 있음
    - 의존하는 인스턴스를 교체하는 것이 비교적 쉽기 대문에 설계를 유연하게 만듬
    - 메시지를 통해 느슨하게 결함됨

--> 따라서 코드 재사용을 위해서는 상속보다는 합성을 선호하는 것이 더 좋은 방법임
