# 2. 객체지향 프로그래밍

## 객체지향 프로그래밍을 향해~~

### 협력, 객체, 클래스

객체 지향은 객체를 지향하는 것

case) 가장 먼저 어떤 클래스가 필요한지 고민하는 것
  - 대부분의 사람들은 클래스를 결정한 후, 클래스에 어떤 속성과 메서드가 필요한지 고민한다.

=> 객체지향의 본질과는 거리가 멀다. **클래스가 아닌 객체에 초점**을 맞춰야 한다.

1. 어떤 클래스가 아닌 어떤 객체들이 필요한지 먼저 고민한다.
   - 클래스는 공통적인 상태와 행동을 공유하는 객체들을 추상화한 것
   - 따라서, 어떤 객체들이 어떤 상태와 행동을 가지는 지 먼저 결정하여 클래스의 윤곽을 잡는다.
   - 객체를 중심에 두는 접근 방법은 설계를 단순하고 깔끔하게 한다.

2. 객체를 독립적인 존재가 아니라 기능을 구현하기 위해 협력하는 공동체의 일원으로 봐야한다.
   - 객체를 협력하는 공동체의 일원으로 바라보는 것은 설계를 유연하고 확장 가능하게 한다.
   - 객체의 윤곽이 잡히면 공통된 특성/상태를 가진 객체들을 타입으로 분류하고, 이 타입을 기반으로 클래스를 구현하자.
  
### 도메인의 구조를 따르는 프로그램 구조

- 도메인: 문제를 해결하기 위해 사용자가 프로그램을 사용하는 분야
- 클래스의 이름은 대응되는 도메인 개념의 이름과 유사하게 지어야한다.
- 클래스 간의 관계도 도메인 개념 사이의 관계와 유사하게 만들어 프로그램의 구조를 이해하고 예상하기 쉽게 만들어야한다.

### 클래스 구현하기

- 클래스의 경계를 구분지어야한다.
  - 클래스는 내부와 외부로 구분되며, 핵심은 어떤 부분을 외부에 공개하고, 감출 지 결정하는 것
  - Why? 경계의 명확성이 객체의 자율성을 보장하기 때문.
 
### 자율적인 객체

```
1. 객체는 상태와 행동을 함께 갖는 복합적인 존재이다.
2. 객체는 스스로 판단하고 행동하는 자율적인 존재이다
```

- 캡슐화: 데이터와 기능을 객체 내부로 함께 묶는 것
- **접근 제어 메커니즘**: 외부에서의 접근을 통제할 수 있는 메커니즘
  - 외부의 간섭을 최소화 하여 객체를 자율적인 존재로 만든다.
- **인터페이스와 구현의 분리**
  - 퍼블릭 인터페이스: 외부에서 접근 가능한 부분
  - 구현: 내부에서만 접근 가능한 부분

### 협력

- 객체는 다른 객체에게 공개된 행동을 수행하도록 **요청**할 수 있다.
- 요청받은 객체는 요청을 처리한 후 **응답**한다.

- 메시지와 메서드
  - 객체간의 상호작용을 위한 유일한 방법 = **메시지를 전송**하는 것
    - 다른 객체에게서 요청이 도착할 때, 객체가 메시지를 수신했다고 함
  - 수신된 메서드를 처리하기 위한 자신만의 방법을 **메서드**라 함

### 상속과 다형성

**상속**
- 기존 클래스를 기반으로 새로운 클래스를 쉽고 빠르게 추가할 수 있는 방법을 제공한다.
- 부모 클래스의 구현은 공유하면서 행동이 다른 자식 클래스를 쉽게 추가할 수 있다.(차이에 의한 프로그래밍)

**상속과 인터페이스**
- 상속을 통해 자식 클래스는 자신의 인터페이스에 부모 클래스의 인터페이스를 포함하게 됨
- **업캐스팅**: 자식클래스는 부모클래스의 수신 가능한 메시지를 수신할 수 있기 때문에, 외부객체는 자식클래스를 부모 클래스와 동일한 타입으로 간주 가능
 
**다형성**
: 동일한 메시지를 전송할 때, 어떤 메시지가 실행될 것인가는 메시지는 수신하는 객체의 클래스가 무엇이냐에 따라 달라지는 것

- 지연/동적 바인딩: 메시지에 응답하기 위한 메서드를 컴파일 시점이 아닌 실행 시점에 결정한다.
  - + 초기/정적 바인딩: 컴파일 시점에 결정되는 것

### 추상화와 유연성

**추상화의 힘**
- 요구사항의 정책을 높은 수준에서 서술할 수 있다.
  - 세부적인 내용을 무시한 채 상위 정책을 쉽고 간단하게 표현 가능
- 추상화를 이용하면 설계가 좀 더 유연해짐

### 합성

- 합성: 인터페이스에 정의된 메시지를 통해서만 코드를 재사용
  - 구현을 효과적으로 캠슐화 가능
  - 의존하는 인스턴스 교체가 쉽기 때문에 설계가 유연해짐
- 상속은 클래스를 통해 강하게 결합되는 데 비해 합성은 메시지를 통해 느슨하게 결합된다.

=> 코드 재사용을 위해서는 상속보다는 합성을 선호하는 것이 좋다.

(다형성을 위해 인터페이스를 재사용하는 경우에는 상속과 합성을 함께 조합해서 사용할 수도 있긴 하다)
