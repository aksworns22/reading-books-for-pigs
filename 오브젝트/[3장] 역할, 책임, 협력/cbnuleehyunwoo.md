# 3장: 역할, 책임 협력

- 객체지향 패러다임에서 관점은 역할(role),  책임(reponsibility), 협력(collaboration)이다.
- 객체지향의 본질이 무엇인가?
    - 협력하는 객체들의 공동체를 창조하는 것
    - 클래스와 상속은 단지 구현 매커니즘이다.
- 역할, 책임 협력이 제자리를 찾지 못한 상태라면
    - 응집도 높은 클래스와 중복 없는 상속 계층을 구현한다 하더라도, 애플리케이션이 침몰한다. (ㄷㄷ)

## 1. 협력

### 영화 예매 시스템 돌아보기

- 객체지향 원칙을 따르는 애플리케이션의 제어 흐름은 어떤 하나의 객체에 의해 통제되지 않고 다양한 객체들 사이에서 균형 있게 분배되는 것이 일반적이다.
- 다양한 객체들이 영화 예매라는 기능을 구현하기 위해
    - 메시지를 주고받으면서
    - 상호작용한다.
- 협력: 객체들이 애플리케이션의 기능을 구현하기 위해 상호작용
- 책임: 객체가 협력에 참여하기 위해 수행하는 로직
- 역할: 객체들이 협력 안에서 수행하는 책임들이 모인 것

### 협력

- 객체지향 시스템 → 자율적인 객체들의 공동체
- 객체는 고립된 존재가 아니다.  → 사회적인 존재
- 객체지향의 세계에서 기능을 구현할 수 있는 유일한 방법 → 협력

> 협력이란 어떤 객체가 다른 객체에게 무언가를 요청하는 것이다..
> 
- 메시지를 수신한 객체는 이를 어떻게 처리하는가? → 메서드
- 객체는 메시지를 처리할 방법을 스스로 선택한다. → 자율적
- 자율적인 객체란…
    - 자신에게 할당된 책임을 수행하던 중에 필요한 정보를 알지 못하거나, 외부의 도움이 필요한 경우
    - 적절한 객체에게 메시지를 전송한다. (협력 요청)
        - 이는 반복될 수 있다. (메시지를 수신한 객체도 또 다른 객체에게 협력 요청..)

### 협력이 설계를 위한 문맥을 결정한다.

- 객체가 가질 수 있는 상태와 행동을 어떤 기준으로 결정해야 할까?

> 어떤 객체도 섬이 아니다. 애플리케이션에서 어떤 객체가 필요하다면 그 이유는 단 하나여야 한다.
> 
- 객체의 행동을 결정하는 것은 객체가 참여하고 있는 협력.
    - 협력이 바뀌면 객체가 제공해야 하는 행동 역시 바뀌어야 한다.
    - 협력은 객체가 필요한 이유와 객체가 수행하는 행동의 동기를 제공
- 따라서, 협력은 객체를 설계하는데 필요한 문맥을 제공한다.

## 2. 책임

### 책임이란 무엇인가?

- 협력이 모두 갖춰진 후 → 협력에 필요한 행동을 수행할 수 있는 적절한 객체 찾기
    - 이때 협력에 참여하기 위해 객체가 수행하는 행동이 책임이다.
- 책임: 객체에 의해 정의되는 응집도 있는 행위의 집합
    - 객체가 유지해야 하는 정보와 수행할  수 있는 행동에 대해 개략적으로 서술한 문장이다.
- 크레이그 라만의 하는 것과 아는 것으로 객체 책임 나누기
    - 하는 것
        - 객체를 생성하거나 계산을 수행하는 등의 스스로 하는 것
        - 다른 객체의 행동을 시작 시키는 것
        - 다른 객체의 활동을 제어하고 조절하는 것
    - 아는 것
        - 사적인 정보에 관해 아는 것
        - 관련된 객체에 대해 아는 것
        - 자신이 유도하거나 게산할 수 있는 것에 대해 아는 것
    - 객체는 자신이 맡은 책임을 수행하는 데 필요한 정보를 알고 있을 책임이 있다.
    - 객체는 자신이 할 수 없는 작업을 도와줄 객체를 알고 있을 책임이 있다.

### 책임 할당

- 정보 전문가 패턴:
    - 책임을 수행하는데 필요한 정보를 가장 잘 알고 있는 전문가에게 그 책임을 할당하는 것
    - 일상 생황에서 도움을 요청하는 방법과도 유사할 것이다.

### 책임 주도 설계

- 어떤 책임을 선택하느냐가 전체적인 설계의 방향과 흐름을 결정
- 책임 주도 설계: 책임을 찾고 책임을 수행할 적절한 객체를 찾아 책임을 할당하는 방식으로 협력을 설계하는 방법
- 책임 주도 설계의 프로세스
    1. 시스템이 사용자에게 제공해야 하는 기능인 시스템 책임을 파악하낟. 
    2. 시스템 책임을 더 작은 책임으로 분할한다. 
    3. 분할된 책임을 수행할 수 있는 적절한 객체 또는 역할을 찾아 책임을 할당한다. 
    4. 객체가 책임을 수행하는 도중 다른 객체의 도움이 필요한 경우 이를 책임질 적절한 객체 또는 역할을 찾는다. 
    5. 해당 객체 또는 역할에게 책임을 할당함으로써 두 객체가 협력하게 한다. 

### 메시지가 객체를 결정한다.

- 메시지를 먼저 식별하고 메시지를 처리할 객체를 나중에 선택하는 것이 중요…
- 객체가 메시지를 선택하는 것이 아니다. 메시지가 객체를 선택하게 한다.
- 왜?
    1. 객체가 최소한의 인터페이스를 가질 수 있게 된다. 
        - 필요한 메시지가 식별될 때까지 객체의 퍼블릭 인터페이스에 아무것도 추가하지 않기 때문에 개체는 애플리케이션에 크지도, 작지도 않은 꼭 필요한 크기의 인터페이스를 가질 수 있다.
    2. 객체는 충분히 추상적인 인터페이스를 가질 수 있게 된다. 
        - 객체의 인터페이스는 무엇을 하는지는 표현해야 하지만 어떻게 수행하는 지를 노출해서는 안 된다.

### 행동이 상태를 결정한다.

- 객체의 존재 이유 → 협력에 참여하기 위해
    - 따라서 객체는 협력에 필요한 행동을 제공해야 한다.
- 실수 1. 객체의 행동이 아니라, 상태에 초점을 맞추는 것
    - 이런 방식은 객체의 내부 구현이 객체의 퍼블릭 인터페이스에 노출되도록 만들기 때문에 캡슐화를 저해한다.
    

## 3. 역할

### 역할과 협력

- 객체는 협력이라는 주어진 문맥 안에서 특정한 목적을 갖는다.
- 역할: 객체가 어떤 특정한 협력 안에서 수행하는 책임의 집합

### 유연하고 재사용 가능한 협력

- 역할이 왜 중요한가?
    - 역할을 통해 유연하고 재사용 가능한 협력을 얻을 수 있기 때문이다.
- 문제를 해결하기 위해서는 객체가 아닌 책임에 초점을 두어야 한다.

> 역할은 다른 것으로 교체할 수 있는 책임의 집합이다.
> 

### 객체 대 역할

- 역할은 객체가 참여할 수 있는 일종의 슬롯이다.
- 오직 한 종류의 객체만 역할에 참여하는 상황이라고 가정해봤을 때 역할이 유용할까?
    - 그럴 때는 간단하게 객체로 간주하는 것이 복잡성을 떨어뜨릴 수 있겠다.
    - 하지만 여러 종류의 객체들이 참여할 수 있다면 역할이다.
